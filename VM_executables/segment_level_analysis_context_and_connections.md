# Segment-Level Analysis: Context and Connections

This document connects a proposed segment-level analysis direction to the existing experimental setup, current results, and methodology files. It establishes WHY segmentation is being considered and WHAT it is meant to add. It does not specify implementation. The critique and plan live in `segment_level_analysis_critique_and_plan.md`.

---

## 1. Anchor: Current Pipeline State

Each workload execution produces one continuous Hamming or cosine temporal trace. The trace is a per-page time series of memory snapshot deltas across the duration of one workload step.

Current offline aggregation:

- window size: 128
- hop: 64
- one feature vector per full run

Per-recording artifacts under each `testN_<subtype>/` folder:

- `streaming.npz` with per-page summary arrays
  - `msc_spectrum` shape `(65, N_pages)`
  - `msc_peak_snr_db_per_page`
  - `plv_per_page`
  - `cepstral_peak_idx_{mean, var, median}`
- `streaming.json` with scalar summaries
- `plv_baseline_aware.json` with PLV stability tier counts
- `meta.json` with frames, pages, window/step sizes, baseline flag

Aggregation order:

1. Per-page sliding windows (128 / 64) reduce time to per-page MSC, PLV, cepstral summaries
2. Per-recording statistics reduce per-page distributions to one scalar feature vector
3. Class-level analysis (Kruskal-Wallis, LDA, silhouette, PCA) operates on those per-recording vectors

The per-page reduction collapses the temporal axis. Once collapsed, no within-run temporal structure is recoverable from `streaming.npz`. This is the structural reason segmentation must operate earlier in the pipeline.

## 3. Current Class-Level Findings (Anchor for Segmentation)

Source: `behavioral_classification_summary.txt`, `stochastic_characterization_summary.txt`, `stochastic_results_scientific_evaluation.md`.

### 3.1 IDLE vs Active separation

- 32 of 41 metrics are statistically significant by Kruskal-Wallis at p < 0.05
- IDLE vs MEM Cohen's d on `snr_high_frac` is -1.468
- IDLE vs IO Cohen's d on `snr_high_frac` is -1.619

### 3.2 MEM vs IO collapse

- All 8 top metrics: pairwise Mann-Whitney p > 0.34 between MEM and IO
- MEM-IO centroid distance is 0.301
- Within-class spread: MEM 1.846, IO 1.594
- MEM mean silhouette is -0.258 (recordings sit inside the IO cluster geometrically)

### 3.3 Within-subtype repeatability (median CV across runs)

| Subtype | median CV | interpretation |
|---|---|---|
| `mem_alloc_touch_pages` | 0.054 | very stable |
| `io_many_files` | 0.048 | very stable |
| `io_seq_fsync` | 0.092 | very stable |
| `io_rand_rw` | 0.189 | stable |
| `mem_stream` | 0.528 | unstable |
| `mem_pointer_chase` | 0.617 | unstable |
| `run_idle` | 0.973 | non-stationary |

### 3.4 Specific run-to-run anomalies that segmentation could probe

`mem_pointer_chase`:

- test4: `snr_mean` = 2.583, `dc_coh` = 0.386
- test16: `snr_mean` = 0.086, `dc_coh` = 0.014
- test28: `snr_mean` = 0.052, `dc_coh` = 0.009
- test40: `snr_mean` = 0.078, `dc_coh` = 0.006

A single subtype shows two regimes across cycles. Either three of four runs collapsed to idle-like behavior, or the workload entered a non-active regime mid-run, or capture artifacts dominated. Full-run aggregation cannot distinguish these.

`mem_stream`:

- test2: `dc_coh` = 0.483
- test14: `dc_coh` = 0.345
- test26: `dc_coh` = 0.134
- test38: `dc_coh` = 0.133

Cycle-1 runs are stronger than cycle-3 and cycle-4 runs. This is consistent with execution-order or thermal effects across cycles, but full-run aggregation cannot localize where in time the signal weakens.

`run_idle` outliers:

- test1 silhouette = -0.481 (first IDLE in cycle 1)
- test3 silhouette = -0.469 (after first MEM workload)

The earliest IDLE recordings sit inside MEM/IO geometry, suggesting residual contamination. Segmentation can test whether this contamination concentrates at the beginning of those IDLE windows.

Source: `confusion_matrix_diagnostic_methodology.md` Section 9.

Provisional confusion structure at n=4 per subtype:

| Direction | Count | True | Predicted |
|---|---|---|---|
| `mem_stream → run_idle` | 1 | mem_stream | run_idle |
| `mem_stream → io_rand_rw` | 1 | mem_stream | io_rand_rw |
| `mem_pointer_chase → mem_stream` | 1 | mem_pointer_chase | mem_stream |
| `io_rand_rw → mem_alloc_touch_pages` | 1 | io_rand_rw | mem_alloc_touch_pages |
| `mem_alloc_touch_pages` | 0 | clean | clean |
| `io_many_files` | 0 | clean | clean |
| `io_seq_fsync` | 0 | clean | clean |

The confusion-matrix methodology already requires every confusion to be assigned ratings across four causes (Section 5 of that file): true behavioral overlap, capture artifact, execution-order contamination, metric inadequacy. Segmentation provides one of the few in-data tools that can begin to discriminate among those causes.

## 5. The Segment-Level Proposal in One Paragraph

Instead of summarizing each Hamming or cosine trace as a single feature vector, split each original trace into `k` contiguous temporal segments along the frame axis. Within each segment, run the existing window/hop pipeline. Aggregate inside each segment to produce one feature vector per segment. The result is a hierarchical dataset: subtype contains runs contains segments. Each segment carries a subtype label inherited from its parent run.

This is contiguous temporal segmentation. It is not random window sampling. It is not random splitting of overlapping windows. It is not data augmentation.

---

## 6. What Full-Run Aggregation Hides and Segmentation Can Reveal

| Phenomenon | Hidden by full-run aggregation | Visible with segmentation |
|---|---|---|
| Warmup vs steady-state | yes | yes |
| Mid-run regime change | yes | yes |
| Workload collapse partway through | yes | yes |
| Cycle-internal phase structure (allocate / touch / release) | yes | yes if segment length matches cycle period |
| Position-of-confusion within a misclassified run | yes | yes |
| Residual contamination at the start of IDLE windows | yes | yes |
| Single-segment outliers vs whole-run outliers | yes | yes |
| Cross-cycle drift within one subtype | partial | yes |
| Capture-side artifacts that cluster in time | no | yes if they cluster within segments |

Segmentation does not create new information. It exposes structure that the per-run summary explicitly averages over.

---

## 7. Why This Is Being Proposed Now

Three pressures from the existing analysis converge:

1. The MEM vs IO collapse and the negative MEM silhouette score indicate that per-run features fail to localize the per-class signal. A segmentation might show that each MEM run has a clean window AND a noisy window that cancel under averaging.

2. The confusion-matrix methodology already mandates separating four causes, but no in-data tool currently exists to begin that separation. Segmentation provides one such tool.

3. The provisional cleanliness designations in the confusion-matrix methodology rest on n=4 per subtype. Without new captures, the only way to inspect within-run stability is to look INSIDE each run.

Segmentation is therefore proposed as a diagnostic methodology, not as a way to manufacture a larger n.

## 8. Direct Connections to Existing Methodology Files

### 8.1 To `vm_workload_executables_purpose_for_ai.md`

That file frames each workload as a controlled stimulus generating a behavioral regime over the full run. It states explicitly:

> Repeated runs of the same subtype should produce similar metric values or similar positions in feature space.

Repeatability is currently measured ACROSS runs. Segmentation extends that question INSIDE a run. Both are repeatability tests at different timescales. A subtype that is stable across runs but unstable across segments tells a different story from one stable in both, or one stable in segments but not across runs.

### 8.2 To `current_workload_test_mini_specs.md`

Each mini-spec lists a `Mechanism stressed` and an `Expected memory pattern`. Many of those expected patterns are inherently temporal:

- `mem_alloc_touch_pages`: "Burst-like allocation phases", "Repeated allocate-touch-release structure"
- `io_seq_fsync`: "Repeated write/sync phases", "Possible periodic flush behavior"
- `io_many_files`: "Small repeated bursts"
- `mem_stream`: "Potentially stronger periodic or quasi-periodic components"

Segmentation enables the analysis to test these temporal-structure expectations directly, rather than only their averaged effects.

### 8.3 To `confusion_matrix_diagnostic_methodology.md`

That file (post-update) defines:

- Section 5 four-cause diagnostic model
- Section 6 provisional language rule (n < 8 to 10 forbids final cleanliness labels)
- Section 7 confusion direction rule
- Section 13 typed expected confusions

Segmentation is the natural extension of Section 5. It is the in-data probe that can begin to assign per-cause ratings:

| Confusion observation under segmentation | Cause it points to |
|---|---|
| All segments of a misclassified run look like the wrong class | true behavioral overlap or metric inadequacy |
| Only early or only late segments look wrong | execution-order contamination |
| Confusion pattern correlates with adjacent capture-pause spikes | capture artifact |
| Misclassified run shows internally consistent pattern that differs from same-subtype runs | true behavioral overlap |

This is a tool, not a verdict. Section 5 rules still apply: ratings remain hypotheses until isolation experiments confirm.
