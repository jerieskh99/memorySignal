SECTION 01

Executive summary

What Plan 04 is

Given the per-workload (iv, W, H) triple frozen by Plans 02 and 03, run an offline change-point detector on each cell's APF trajectory; route the per-cell metric through a family-aware dispatcher (phasic uses windowed F1 against the phase-marker ground truth; steady uses a spurious-boundary count); and emit a per-workload acceptance verdict plus a per-cell breakdown. The sweep reads the existing 132 v3 cells; zero new captures. Output: plan04_segments.csv, plan04_summary.json, and plan04_recommendation.json, the last consumed by a new validator claim C8.

What changes vs the design doc
  • Δ-1 · pick windowed CUSUM on the W=8 APF mean series (rationale in §07 dispute 1)
  • Δ-2 · family-aware dispatch: phasic uses windowed F1, steady uses spurious-boundary count (reuses plan02_run._classify_workload)
  • Δ-3 · add C8 operational validator claim · gates analysis-ready once Plan 04 has run
  • Δ-4 · keep detect_boundaries_diff as a backward-compatible fallback behind --detector legacy; Plan 02 v1/v2 paths unchanged
  • Δ-5 · synthetic + real test fixtures (step-function ramp, constant, single-phase, real v3 fault_only cell)
  • Δ-6 · per-workload boundary tolerance: 1 snap for ransom_batched, 2 snaps for ransom_seq and hashtable, 0 for steady (no boundaries expected)
Net effect. Plan 04 lands as approximately 420 LOC of new code across four new files plus targeted edits in two existing ones. Zero new VM captures. Approximately 20 seconds of sequential compute on the 132-cell v3 bundle. C8 closes the boundary-detection loop: phasic cells get a window-tunable F1; steady cells get a "did the detector hold its peace?" check.
Bound surfaced. Phase markers are only emitted by a subset of workloads (the four ground-truth-rich phasic tests plus mem_pagefault_density_v2:fault_only and app_hashtable_intensive_v2); the remaining workloads cannot compute F1. Plan 04 explicitly documents this and falls back to stationarity-ratio + spurious-count for marker-less phasic workloads, mirroring Plan 03's best-feasible pattern. A true F1 lift for marker-less cells is out of scope (it would require either a new workload-side --phase-markers capability or an unsupervised reference, both beyond Plan 04).

SECTION 03

Baseline · the design doc as written

What docs/tuning_plans/04_k_segmentation_tuning_plan.md proposed

Sweep the segmenter's segment count k per workload and min_windows_per_segment across {10, 25, 50, 100}, holding the Plan 02 interval and Plan 03 window/hop fixed. Use F1 against phase markers for ground-truth-rich workloads; for the rest, use within-vs-across segment variance ratio (target ≤ 0.5). Estimate ~10 host CPU-hours, zero new captures.

Concrete spec
  • Independent variables: k per workload, min_windows_per_segment ∈ {10, 25, 50, 100}
  • Dependent variables: F1, per-segment stationarity ratio, segmenter rejection rate
  • Acceptance: F1 ≥ 0.8 on the four marker-rich workloads, rejection rate < 5 %, stationarity ratio ≤ 0.5, forced-k=1 does not split spuriously
  • Implementation surface: offline_step_metrics.py:_MIN_WINDOWS_PER_SEGMENT_DEFAULT made configurable (~20 LOC), per-test metadata schema (~10 LOC), small segmenter_audit.py driver (~150 LOC)
  • Open follow-ups: adaptive k selection, online segmentation

What the design doc did NOT account for

  • The 1-D streaming APF series has no segmenter to tune. offline_step_metrics.py was the segmenter for the 2-D dump-matrix path. v2 / v3 use B+3.1 streaming and never write the matrix; consequently the existing segmenter has no input to operate on. Plan 04's actual surface is plan02_metrics_per_cell.detect_boundaries_diff, not the offline segmenter, and the substitution is what unlocks Plan 03's choice of W and H downstream.
  • F1 in v2 was already a window-independent stub. detect_boundaries_diff takes only the trajectory; it has no W or H argument and produces identical boundaries for every (W, H) combo. Plan 03 §06 names this the "subtlety that mattered" and worked around it by switching the ransom defining metric to cepstral SNR. Plan 04's central job is to replace the stub with a windowed detector so the F1 lever becomes real.
  • The k question is mostly moot at v3. Of the eleven v3 workloads, eight have ground-truth k = 1 (single-segment steady or single-phase phasic). Of the three multi-segment workloads, only sandbox_ransom_batched (k = 5) and app_hashtable_intensive_v2 (k = 2) emit phase markers usable for F1; mem_pagefault_density_v2:fault_only emits markers but at k = 2. Sweeping k across {1, 2, 5} is the right shape, not the doc's wider scan.
  • min_windows_per_segment ∈ {10, 25, 50, 100} is infeasible at the Plan 03 grid. At W=8, H=4 on a 60-second cell with iv=500 ms, the cell produces approximately 16 snaps and approximately 3 windows; the smallest level in the doc (10 windows) exceeds the total available. The level set must be re-baselined against Plan 03's n_windows.
  • Backward compatibility is unstated. The design doc does not address how the new segmenter coexists with the v2/v3 cells whose per-cell JSON already records an f1_phase computed from the stub. Plan 04 must preserve that lookup path so the v2 baseline regression guard from Plan 03 §08 keeps working.

Objective alignment

What we're trying to achieve

Make the boundary-detection lever actually move when Plan 03 changes (W, H); route per-family acceptance so steady workloads are not unfairly judged by a metric (F1) that has no semantics for them; ship a single artifact that downstream consumers can read.

Does the spec meet it?

Yes within the marker-coverage regime. A windowed CUSUM on the W=8 rolling mean is window-sensitive by construction (the input series itself depends on W and H), it has a single tunable knob (k = 2, the standard 2σ rule), and it can be unit-tested on synthetic step functions. The marker-less cells fall back to a spurious-boundary count, which mirrors Plan 03's best-feasible pattern.

Structural risks

  • R1 · marker coverage is thin. Of the eleven v3 workloads, only five emit phase markers usable as ground truth. The F1 evidence base is therefore approximately 5 × 4 = 20 cells per duration stratum, before further filtering by n_windows floor. Mitigation: report per-workload F1 with bootstrap CIs and an explicit "marker-less" carve-out in the recommendation JSON.
  • R2 · steady acceptance is asymmetric. A steady workload's score lives on a "lower is better" spurious-count axis with no upper-bound symmetry vs F1. Combining the two in a single C8 verdict requires per-family gates, not a pooled metric. Mitigation: G1 (phasic F1) and G2 (steady spurious) are evaluated independently; C8 PASS requires every applicable workload to clear its own gate.
  • R3 · window-centre alignment. CUSUM emits boundaries in window index space; the F1 scorer wants snap indices to match the phase-marker emission space. The transform snap_idx = window_idx × H + W // 2 introduces a constant W/2 = 4-snap bias for every prediction at W=8. Mitigation: bake the centring into the dispatcher and document it; the per-workload boundary tolerance absorbs the residual.
  • R4 · CUSUM hyperparameter (k). The threshold k on the cumulative sum is the only tuning knob in the v1 plan. The rule of thumb is 2σ, but on a robust MAD scale the effective level changes. Mitigation: sweep k ∈ {1.5, 2.0, 2.5} in the test harness, keep k = 2.0 as the default; allow per-workload override in the recommendation JSON.
  • R5 · cherry-picking on F1. Plan 03 §11 already flagged this for the analyzer sweep. Plan 04 inherits the same risk for the detector sweep. Mitigation: report median F1 per workload AND P(F1 ≥ 0.67) jointly; do not aggregate across families.

5.1 · plan04_change_point.py

What it does

Pure function. Takes a 1-D APF trajectory plus (W, H) plus an optional CUSUM threshold k; returns a sorted list of snap-index boundaries. No I/O. Easy to unit-test on synthetic ramps and step functions.

Algorithm spelled out
  • Compute rolling mean μ_i over each W-wide window with hop H → a series of length n_windows
  • Robust scale estimate σ̂ = MAD(μ) × 1.4826 (median absolute deviation, scaled to match the normal)
  • Two-sided CUSUM: S_i⁺ = max(0, S_{i-1}⁺ + (μ_i − μ̄) − k σ̂); S_i⁻ the symmetric negative version
  • Peak detection on |S⁺| + |S⁻| with threshold 5 σ̂; collapse consecutive peaks within W of one another
  • Map window-index peaks back to snap index via snap_idx = window_idx × H + W // 2
  • Single tuning knob: k (default 2.0, the standard 2σ CUSUM rule)

5.2 · plan04_dispatch.py

What it does

Family-aware scoring. Reuses plan02_run._classify_workload as the source of truth for phasic vs steady (no duplicate logic). For phasic workloads with markers, calls f1_score at a per-workload tolerance. For steady workloads, simply counts predicted boundaries: a steady workload should produce zero.

Per-workload boundary tolerance (Δ-6)
  • sandbox_ransom_batched: tolerance 1 snap (sharp transitions)
  • sandbox_ransom_seq: tolerance 2 snaps (per-file granularity is fuzzier)
  • sandbox_ransom_selective: tolerance 1 snap
  • app_hashtable_intensive_v2: tolerance 2 snaps (build/probe transition is gradual)
  • mem_pagefault_density_v2:fault_only: tolerance 2 snaps (decay tail blurs the boundary)
  • steady workloads: tolerance = 0 (no boundaries expected; count any prediction as spurious)

5.3 · plan04_sweep.py

What it does

CLI driver. Reads manifest, iterates cells, reads (W, H) from plan03_recommendation.json, calls the detector and dispatcher, streams output to CSV with per-row reproducibility hash. Approximately 140 LOC.

CLI surface
  • --cells-dir · v3 cells path
  • --manifest · manifest.csv path
  • --plan03-rec · path to plan03_recommendation.json
  • --output-csv · per-cell rows
  • --output-json · aggregated summary (calls aggregator)
  • --detector default cusum; legacy selects detect_boundaries_diff for Δ-4 backward compat
  • --cusum-k default 2.0
  • --workload-filter regex
  • --workers default 1
  • --seed default 0 (for any bootstrap inside)

5.4 · plan04_aggregate.py

What it does

Reads segments.csv. Groups by workload. Applies gates G1-G4. Picks per-workload verdict (PASS / best-feasible / FAIL). Writes recommendation.json + markdown table. Includes a regression check against the legacy detect_boundaries_diff output stored in the per-cell JSON for marker-rich workloads.

Decision rule (DS approved)
  • Phasic workloads with markers: gate G1 fires when median F1 ≥ 0.67 AND per-cell P(F1 ≥ 0.50) ≥ 0.75
  • Phasic workloads without markers: gate G3 fires when n_predicted is plausible (1 ≤ pred ≤ k_expected + 2)
  • Steady workloads: gate G2 fires when median spurious count ≤ 1 AND P(spurious = 0) ≥ 0.5
  • Regression guard (Δ-4): on marker-rich workloads, new median F1 must not fall below legacy F1 minus 0.10; otherwise tag regression_vs_legacy=true and fall back to detect_boundaries_diff

SECTION 07

Key disputes + decisions

Dispute 1 · Which change-point detector

The fight

ML opened with five candidates: CUSUM, BOCPD, PELT, median + MAD on the rolling derivative, and a small CNN. SA and DS argued the budget question: with approximately 20 marker-rich cells per duration stratum, any detector with more than one knob is over-parameterised. EN seconded on implementation cost: BOCPD requires a hazard prior, PELT a penalty term and a cost-model selection, the CNN a training set we do not have.

The resolution

Windowed CUSUM. One knob (k, default 2.0 from the standard 2σ rule). Implementation is approximately 30 LOC over the existing plan03_metric_kernel windowing math. The robust-scale variant uses MAD × 1.4826 rather than the sample standard deviation, which handles the bimodal APF distribution without re-tuning. Per-workload override of k remains supported in the recommendation JSON for future use but is not exercised in v1.

Dispute 2 · Family-aware vs unified score

The fight

EE wanted a single per-cell metric so the aggregator can pool across all 132 cells. DS pointed out that steady workloads produce zero ground-truth boundaries, so F1 is undefined (TP, FP, FN all touch zero denominators). XD proposed the split: phasic → F1, steady → spurious count.

The resolution

Family-aware dispatch via plan02_run._classify_workload. Phasic with markers uses windowed F1 against the parsed workload_stderr.log markers; phasic without markers falls back to predicted-count plausibility; steady uses spurious-count (lower is better). C8 PASS requires every applicable workload to clear its own family-specific gate. No pooled score is reported, on principle.

Dispute 3 · Per-workload boundary tolerance

The fight

XD argued that ransom_batched transitions are sharp (binary on / off encryption) while ransom_seq transitions are fuzzy (per-file granularity bleeds across snaps) and hashtable transitions are gradual (build / probe shift over multiple seconds). A single global tolerance would either be too tight for the gradual workloads or too loose for the sharp ones.

The resolution

Per-workload tolerance baked into plan04_dispatch.py: ransom_batched and ransom_selective at 1 snap; ransom_seq, hashtable, and fault_only at 2 snaps; steady at 0 snaps (no boundaries permitted). At iv=500 ms, 1 snap is 500 ms of guest time, which is a reasonable epsilon for the sharp workloads.

Dispute 4 · Backward compatibility of detect_boundaries_diff

The fight

EN wanted to delete the stub and replace it with the CUSUM detector wholesale, on the grounds that two boundary detectors in the codebase invite drift. EE pushed back: the existing per-cell JSON records an f1_phase field computed from the stub, and Plan 03's v2 regression guard reads it; deleting the stub would silently break that lookup.

The resolution

Keep the stub verbatim. Add the CUSUM detector under a new module. Select between them via --detector cusum|legacy on the sweep CLI, with cusum as default. The per-cell JSON's f1_phase field continues to record the legacy F1 from the stub; the new Plan 04 recommendation file records the CUSUM F1 separately. Plan 03's regression guard keeps working.

Dispute 5 · Regression guard against legacy stub

The fight

EE proposed a regression guard: if the new CUSUM F1 falls more than 0.10 below the legacy stub F1 on a marker-rich workload, tag regression_vs_legacy=true and refuse to promote. DS pointed out that the stub's bimodal F1 distribution (mass at 0 and 1) means the median can swing 0.20 from a single-cell flip, so the threshold needs to be loose.

The resolution

Threshold of 0.10 on the median, evaluated per workload (not pooled). When the guard fires, the aggregator emits regression_vs_legacy=true AND passes_acceptance=false AND falls back to the legacy stub in the recommendation for that workload. C8 then fails for that workload, surfacing the regression to the operator for sign-off.

Dispute 6 · Test fixtures · synthetic vs real

The fight

EN wanted synthetic-only tests (cheap, fast, deterministic). DS argued that synthetic step functions do not exercise the bimodal APF noise floor. XD wanted real v3 fixtures committed to the repo (a small subset).

The resolution

Both. Synthetic fixtures: (a) step 0 → 1 → 0, (b) constant, (c) ramp, (d) noisy step. SECTION 10

Statistical design

Test framework

  • Within-workload comparison of new vs legacy F1: Wilcoxon signed-rank, paired per cell. Non-parametric because F1 is bounded [0, 1] with mass at exactly 0 and 1; the stub and CUSUM both inherit the bimodal shape.
  • Steady spurious-count comparison: Mann-Whitney U on the n_spurious vector vs the expected-zero null; one-sided (lower is better).
  • Bootstrap CIs: BCa at 10 000 iterations, stratified resample within (iv, d) blocks. Reported as the headline metric in recommendation.md.
  • Per-workload only. No pooling across families: F1 and spurious-count are not commensurable.

Sample size + power · honest accounting

Power is limited. v3 effective n per (workload, duration) after replicate consolidation: 4 cells. Pooled across durations: 12 cells per workload. Against an effect size of Cohen's d ≈ 0.40 (F1 delta of approximately 0.15), Wilcoxon power at α = 0.05 is approximately 31 % (per workload). Claims about CUSUM improving on the diff stub must therefore be stated as effect-size estimates with confidence intervals, not as significance tests; the recommendation's delta_vs_legacy field is the ranking statistic, with the BCa CI reported alongside.

Multiple-comparisons handling

Holm-Bonferroni FWER control per workload, not pooled. Five marker-rich workloads with one comparison each (new vs legacy F1) = 5 tests; correction floor at α / 5 = 0.01. Steady workloads contribute six one-sided spurious-count tests against the null of zero; correction floor at α / 6 ≈ 0.0083. The C8 decision propagates downstream (false positives have physical cost in the thesis chapter), so FWER is the right control, not FDR.

Replicate non-independence

Four replicates per (workload, duration) share VM state and host time-of-day, the same constraint Plan 03 §10 documented. Mitigation: paired tests within (workload, duration) before aggregating; report per-cell F1 distributions, not pooled means; cell-level breakdowns kept in segments.csv for downstream audit.

Bimodality-aware reporting

F1 is bimodal in the v3 data: cells either match all markers (F1 = 1.0) or miss most (F1 ≤ 0.5). The mean is therefore misleading. Plan 04 reports the median AND P(F1 ≥ 0.67) AND the histogram (binned 0, [0, 0.25), [0.25, 0.50), [0.50, 0.75), [0.75, 1.0), 1.0) in the recommendation markdown.

[BCa 95 % CI: ?, ?] P(F1 ≥ 0.67) = ? Delta: ? [BCa 95 % CI: ?, ?] Wilcoxon signed-rank: W = ?, p = ? (Holm-corrected p = ?) Bootstrap: 10 000 iterations, BCa, stratified by (iv, d)
