Snapshot Interval — Q&A

Companion to SNAPSHOT_INTERVAL_TUNING_STUDY.md and SNAPSHOT_INTERVAL_QA.md. Explains the corrected timing model for the QEMU/pmemsave capture pipeline.

Main takeaway

The pipeline has three different time axes. An earlier study conflated them and produced wrong conclusions about what intervalMsec buys you.

A · Guest-running interval

Wall-clock gap as seen by the guest's own monotonic clock. Frozen during pmemsave. Sets analysis frame spacing.

B · Host wall-clock per snapshot

Real host time per frame: suspend + pmemsave + resume. About 2.5 s for 1 GiB RAM on SSD. Sets production rate.

C · Analysis frame spacing

What the analyzer uses for cepstra/MSC/PLV time axes. Equals Axis A, not Axis B.

D · Experiment throughput

total_snapshots × Axis-B. The wall-clock cost of the experiment. What makes it "feel slow".

Bottom line. intervalMsec sets Axis A, not Axis B. With intervalMsec = 100, the analyzer sees ~100 ms of guest activity between consecutive frames, even though each frame takes ~2.5 s of host wall-clock to produce. The two are decoupled.

Core Validation Questions

The ten questions used to verify the corrected timing model against the actual producer script.

1. Does the current capture loop pause the VM during pmemsave?

Yes. The producer (capture_producer_qemu_pmemsave.sh) calls virsh suspend before every snapshot and virsh resume after it. The exact order (lines 91–164):

virsh suspend       (line 92)   ← VM goes paused
wait domstate=paused (line 98)
pmemsave 1 GiB      (line 106)  ← VM frozen, dump to disk
sleep 0.5           (line 114)  ← flush
enqueue prev/curr   (line 138)
virsh resume        (line 154)  ← VM goes running
wait domstate=running (line 156)
sleep intervalMsec  (line 161)  ← VM running
[next iteration]

The VM is paused from line 92 to line 154 — across the pmemsave window.

2. Does intervalMsec represent guest-running time, host wall-clock, or something else?

Guest-running time between snapshots (Axis A). The sleep intervalMsec/1000 (line 161) happens after virsh resume, while the VM is running. The next iteration's virsh suspend follows that sleep.

So between two consecutive pmemsave moments, the guest executes for approximately intervalMsec of its own time, plus a few milliseconds of bash bookkeeping.

3. Where exactly does sleep happen relative to resume/suspend?

After resume, before the next iteration's suspend. The sleep is the only part of the loop where guest time advances. Everything else either (a) happens before resume (so the VM is still paused) or (b) happens after resume but is so brief it adds only a few ms.

There is one additional sleep 0.5 (line 114) inside the paused window — a flush wait for pmemsave. That sleep happens while the VM is paused, so it does not consume guest time; it adds 0.5 s to Axis B.

Clarification on "inflates Axis B". Axis B is the additive sum of all host-side wall-clock costs per snapshot. The 0.5 s flush is one term in that sum; it is not double-counted anywhere. The choice of "inflates" was informal. The flush is also arguably redundant: qemu-monitor-command pmemsave returns synchronously after writing all bytes, so the 0.5 s belt-and-braces sleep is a safety margin from when this path was first prototyped, not a correctness requirement. Whether to keep it is a separate decision (see docs/tuning_plans/02_interval_tuning_experiment.md); the accounting itself is correct.
4. During pmemsave, does the guest workload progress or is it frozen?

It is frozen. QEMU stops the VCPUs and pauses the guest TSC during virsh suspend. Inside the VM, clock_gettime(CLOCK_MONOTONIC) does not advance. A workload thread blocked on nanosleep(100ms) does not wake during pause — it wakes at guest-time T + 100ms, which on the host wall-clock may be T + 100ms + (2 s of pmemsave + bookkeeping).

From the workload's perspective, the snapshot is instantaneous. The delta between consecutive snapshots reflects what the workload changed during ~100 ms of its own execution — exactly the thing the analyzer should measure.

5. Should analysis frame spacing be interpreted as guest-running interval or host wall-clock time?

Guest-running interval. The deltas computed by the consumer measure guest memory change per intervalMsec of guest execution. Cepstral peak indices, MSC frequencies, PLV time axes — all of these should be reported using Δt_frame = intervalMsec, not the host wall-clock per snapshot.

If a downstream consumer converts frame index k to seconds, it should multiply by intervalMsec / 1000, not by host_dt_mean_s.

6. What timing values must be logged to prove this?

Six per-snapshot timestamps from the producer (clock_gettime(CLOCK_REALTIME) on the host):

TimestampWhere in the loop
t0_before_suspendBefore virsh suspend (line 92).
t1_after_suspendAfter domstate == paused (line 98).
t2_pmemsave_startBefore invoking the pmemsave monitor cmd (line 106).
t3_pmemsave_endAfter the monitor cmd returns.
t4_before_resumeBefore virsh resume (line 154).
t5_after_resumeAfter domstate == running (line 156).

Derived per-snapshot:

  • host_dt = next_t0 − t0 — Axis B.
  • pmemsave_dt = t3 − t2.
  • vm_pause_dt = t5 − t0 — host wall-clock the VM was paused.
  • guest_run_dt = next_t0 − t5 — Axis A on the host side (equals sleep intervalMsec/1000 plus bash overhead when no backpressure).

Optional: probe the guest's CLOCK_MONOTONIC over SSH at t5 and at next_t0; the delta should match guest_run_dt to within a few ms.

8. Which parts are incorrect or overstated?
These claims were tied to confusing Axis B with Axis A. They are corrected in the revised study.
  • "Effective sampling frequency is closer to 0.2–0.4 Hz, not 10 Hz" — wrong if read as frame spacing. That's Axis B (production rate), not Axis A (frame spacing).
  • "Realistic effective Δt ≈ 3 s" — wrong axis when applied to spectral analysis.
  • Section 3's "realistic" sub-tables showing 0 windows for any 10-min run — wrong; window count is set by guest duration ÷ intervalMsec, not by wall-clock cost.
  • "k=2 segmentation requires 5.4 hr at realistic Δt" — confused 10.9 min of guest time with ~4.5 h of wall clock.
  • "window=128 covers ~6.4 minutes of wall clock" — wrong; window is 128 × intervalMsec of guest time.
  • "sandbox_ransom_seq is infeasible at 1 GiB pmemsave" — wrong as a resolution claim.
  • "vm_pause_fraction ≤ 0.30 because of guest/host wall-clock divergence" — wrongly framed.
9. How should the interval/duration tuning plan be revised?
  1. Rename "Δt" → "Δt_frame" everywhere, defined as intervalMsec.
  2. Add a parallel host-wall-clock-cost view for every guest-time plan, so the throughput-vs-resolution trade-off is visible.
  3. Reframe family recommendations as "what guest-time resolution does each family need?" — wall-clock cost is set by guest duration, not by interval.

Option B per-family profile still stands; profiles select resolution (intervalMsec) and guest duration jointly. Wall-clock cost falls out of those choices.

10. What does this imply for window=128, hop=64, and segment-level analysis?
  • One window covers 128 × intervalMsec of guest time.
  • One hop covers 64 × intervalMsec of guest time.
  • Spectral peaks at k / (window × intervalMsec) Hz in guest time.
  • k-segment segmentation needs ≥ 50 k windows ≈ 3264 k frames ≈ 3264 k × intervalMsec of guest time.
  • Wall-clock cost ≈ 3264 k × 2.5 s ≈ 2.3 k hours, independent of intervalMsec.

So k=2 segmentation at any intervalMsec costs ≈ 4.6 host hours regardless of whether the chosen interval is 100 ms or 1 s. The interval controls resolution, not cost.

Practical Understanding Questions

The plain-language version of the same ideas.

11. Is my 100 ms interval actually wrong?

No. It is a legitimate choice for guest-time resolution — you get 100 ms frame spacing in the analyzer. What is "wrong" is the expectation that 100 ms means anything about how long the experiment takes. At 1 GiB RAM you pay ~25 s of host wall-clock per second of guest data.

12. Does pmemsave time count as workload execution time?

No. While pmemsave runs, the VM is paused; the guest's monotonic clock does not advance. Workload threads do not run; their internal timers do not fire.

13. What is the difference between guest-running time and host wall-clock time?

Guest-running time advances only while the VM is running. Host wall-clock advances always. During pmemsave the VM is paused for ~2 s of host wall-clock, but the guest sees 0 ms.

A 60 s guest workload at intervalMsec=100ms makes 600 snapshots; at 2.5 s host cost per snapshot, the host pays ≈ 25 minutes to collect 60 s of guest activity.

Why does it feel like the experiment is much longer than the guest run? Because each snapshot is ~25× more expensive than the gap it covers. With intervalMsec=100ms and a ~2.5 s pmemsave, every 100 ms of guest execution costs ≈ 2500 ms of host wall-clock. Over a 60 s guest run, that ratio compounds into a 25-minute host run. The guest log still shows 60 s of activity (correct, in guest time); the operator's wall clock shows 25 minutes (also correct, in host time). The two are not in conflict — they are the two axes from the main takeaway.

14. If the VM is paused during pmemsave, what does one frame mean?

It is a snapshot of guest RAM taken at a particular instant of guest execution. The delta between consecutive frames is "what changed in guest memory during the intervalMsec of guest time between the two pmemsave moments". From the guest's perspective, ~100 ms of execution.

15. Why does the experiment feel slow if the guest only runs 100 ms between samples? Why does it take hours for a few minutes of guest activity?

Because producing the snapshot is expensive. The host has to suspend the VM, write 1 GiB to disk, resume the VM, run the delta, append to the run matrix, optionally compute streaming metrics — all of that takes ~2.5 s. The "slowness" is Axis B (production rate), not Axis A (sampling resolution).

Two levers move host wall-clock:

  1. Snapshot count — set by guest_duration / intervalMsec. Bigger intervalMsec → fewer snapshots → less wall-clock.
  2. Per-snapshot cost — set by Axis B, dominated by 1 GiB pmemsave and disk bandwidth. Smaller VM RAM → cheaper per snapshot.
16. Does increasing intervalMsec make each snapshot cheaper? Will the same guest run finish faster because the VM is paused fewer times?

No to the first; yes to the second. Each pmemsave still writes 1 GiB and each suspend/resume cycle has the same latency — per-snapshot cost (Axis B) is unchanged. But a bigger intervalMsec means fewer snapshots per guest second, so fewer pauses, fewer pmemsave dumps, fewer disk writes, and a proportionally shorter total host wall-clock for the same guest duration.

The cost is in guest-time resolution: 1 s frame spacing cannot see rhythms faster than ~2 s (Nyquist), and windows become 10× coarser in guest time. The right joint choice of (intervalMsec, window, hop) depends on the rhythm of interest — see docs/tuning_plans/02_interval_tuning_experiment.md and docs/tuning_plans/03_window_hop_tuning_experiment.md.

17. What does increasing intervalMsec actually change?
  1. Guest-time spacing between samples grows. Analyzer sees coarser temporal resolution — rhythms faster than 2 × intervalMsec become invisible (Nyquist).
  2. Snapshot count for the same guest duration drops proportionally, so the experiment finishes in proportionally less wall-clock time.

So intervalMsec is a throughput-vs-resolution knob, not a "how-slow-is-each-snapshot" knob.

18. How many frames do I need for window=128 and hop=64? (deferred — revisit after window/hop tuning)

Minimum 128 (one window). For useful sliding-window statistics, the offline analyzer's default expects at least 50 windows per segment, which is (50 − 1) × 64 + 128 = 3264 frames per segment. For k segments, ≈ 3264 k frames.

20. What does it mean in host collection time?

≈ 128 × wallclock_per_snapshot ≈ 128 × 2.5 s ≈ 5.3 min, almost independent of intervalMsec. Each window costs roughly the same wall-clock to produce regardless of which guest-time interval you chose.

21. How does this affect segment-level analysis?

Segment-level analysis is bounded by frame count, not directly by guest time or wall-clock time. To get k segments × 50 windows × ~64 hop ≈ 3264 k frames, choose:

  • a small intervalMsec if you need fine guest-time resolution within each segment, and accept the wall-clock cost; or
  • a larger intervalMsec if your phenomenon is slow in guest time, and pay less wall-clock.

In both cases the host pays roughly the same wall-clock to produce the segmenter input.

22. How should I choose 100 ms vs 250 ms vs 500 ms?
  1. What is the fastest guest-time rhythm I care about? Need intervalMsec ≤ rhythm / 2 for Nyquist; ≤ rhythm / 4 to see the peak cleanly in cepstra.
  2. What is the slowest guest-time rhythm I care about? Need window × intervalMsec ≥ rhythm (otherwise the window cannot contain a full period).
  3. What is my wall-clock budget? Compute snapshots × 2.5 s = guest_duration / intervalMsec × 2.5 s. If too high, increase intervalMsec until it fits.

The six per-snapshot timestamps in question 6. Plus per-run aggregates: guest_dt_mean_s, guest_dt_std_s, host_dt_mean_s, vm_pause_fraction, backpressure_events.
