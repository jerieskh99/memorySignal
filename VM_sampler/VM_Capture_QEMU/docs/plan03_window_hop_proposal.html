Plan 02 v2 published 90 lossless APF trajectories. Plan 03 sweeps analyzer-side window/hop on those same files. Zero new captures. Team review reveals the design doc's window range {32-512} is infeasible at v2 snap counts (0/90 cells admit W=128). SECTION 01

Executive summary

What Plan 03 is

Given the per-workload iv chosen by Plan 02 (ransom = 500 ms, workingset = 250 ms), find the analyzer's sliding-window size and hop that maximize signal-to-noise on each workload's defining metric. Sweep is offline; reads the 90 existing v2 apf_trajectory.jsonl files. Output: a per-workload recommendation table + a CSV of per-(cell, window, hop) metrics.

What changes vs the design doc
  • Δ-1 · narrow window range from {32, 64, 128, 256, 512} to {8, 16, 32, 64} (v2 snap counts cap at 73)
  • Δ-2 · re-baseline acceptance thresholds against measured v2 (F1 ≥ 0.67 per cell + median ≥ 0.85; CV ≤ 0.05 at d ≤ 120 s)
  • Δ-3 · adopt 1-D APF path via [T, 1] reshape (existing cepstrum module is reused as-is)
  • Δ-4 · introduce C7 operational validator claim + plan03_recommendation.json artifact
  • Δ-5 · v2 regression guard: winner must not degrade ransom median F1 below 0.95 or push workingset CV above 0.05
Net effect. Plan 03 lands as ~290 LOC across 3 new files. Zero new VM captures. ~15 seconds sequential compute. C7 closes the analysis-ready loop: cells go from low_window at canonical (128, 64) to passing once Plan 03 publishes a v2-feasible recommendation.
Blocker surfaced. v2's measured snap-count distribution (5-73 per cell, median ~22) constrains every window-size decision. Plan 03 cannot recover the spectral-coverage requirement of the design doc (W × iv ≥ 4 × workload_rhythm = 50 s for ransom, requiring W ≥ 100 at iv=500 ms). Proposal documents this as a known constraint of the B+3.1 streaming regime; full spectral coverage is a future Plan-05 question once snap throughput improves.

SECTION 03

Baseline · the design doc as written

What docs/tuning_plans/03_window_hop_tuning_experiment.md proposed

Sweep window ∈ {32, 64, 128, 256, 512} × hop ∈ {window/4, window/2, window} on the existing Plan 02 captures, hold iv fixed per workload, compute cepstral peak width + spectral leakage + per-window stationarity + the workload's defining metric (F1 for ransom, CV for workingset, cepstral periodicity for sqlite OLTP). Pick the smallest window that passes acceptance for ALL workloads simultaneously.

Concrete spec
  • 5 windows × 3 hop ratios × 3 workloads = 45 analyzer cells per family
  • Wall-clock cost: ~10 host CPU-h, zero new captures
  • Acceptance: stationarity ≥ 90 %, spectral coverage W × iv ≥ 4 × workload_rhythm, F1 ≥ 0.8 / CV ≤ 0.15, hop ≤ window/2
  • ~80 LOC sweep driver, manifest gains window_size + step_size columns
  • Risk register at bottom of doc flags cherry-picking + spectral artifacts at W < 32

What the design doc did NOT account for

  • v2 used B+3.1 streaming, which writes 1-D APF traj only. Doc presumed access to raw dump matrices for cepstral analysis. v2 deletes dumps after per-pair diff (helper Δ-1). Mismatch must be resolved before sweep can run.
  • v2 snap counts ranged 5-73 per cell. Doc's W range {32, 64, 128, 256, 512} is infeasible against this distribution (0/90 cells admit W=128).
  • v2 F1/CV distributions are now measured (median F1 = 1.0 at iv=500 ransom; median CV = 0.022 at iv=250 workingset). Doc's loose thresholds (0.8 / 0.15) sit at or below v2 baseline → would trivially pass without proving Plan 03 improves anything.
  • sqlite OLTP not captured in v2. Doc lists it as third workload. Adding requires new capture (3-5 h) OR scope reduction.
  • Stationarity = StabilityValidator is misleading. That class was built for PLV/MSC on 2-D [T, N] page matrices, not 1-D APF stationarity. Reusing its name in the doc creates confusion.

Objective alignment

What we're trying to achieve

Pick a single (window, hop) per workload such that downstream analyzers (segmenter, cepstrum) get the cleanest signal achievable on v2's APF trajectories, without re-capturing.

Does the spec meet it?

Yes within the snap-count regime. Spectral coverage requirement (W × iv ≥ 4 × workload_rhythm) cannot be met at v2 snap counts without a producer-side throughput fix; Plan 03 explicitly defers that to a future plan and ships a best-feasible (W, H) recommendation with the constraint documented.

Structural risks

  • R1 · snap-count feasibility ceiling. Every v2 cell has snaps < 128. W ≥ 64 is dead-on-arrival for >90 % of cells. Mitigation: narrow range to {8, 16, 32, 64} with explicit "infeasible cells" reporting per combo.
  • R2 · 1-D cepstrum is mathematically valid but uses the module's degenerate code path. compute_cepstrum on [T, 1] works, but compute_cepstral_peak's default min_quef_idx=10 discards the entire spectrum for T < 20. Mitigation: override to max(1, T//8).
  • R3 · stationarity is not StabilityValidator. Plan 03 ships its own stationarity_per_window inline (rolling mean / std-of-means ratio). Naming alignment with design doc requires explicit clarifying note.
  • R4 · cherry-picking on F1. Design doc §8 already warns. Objective function MUST penalize combos that pass on long-d cells while failing short-d cells (DS feasibility penalty).
  • R5 · acceptance-threshold drift. The doc's F1 ≥ 0.8 sits at v2's mean (50 % expected pass). Loose threshold = automatic "pass" regardless of (W, H). Mitigation: tighten to v2-grounded floors per the ML+EE proposal.

5.1 · plan03_metric_kernel.py

What it does

Pure function. Take a 1-D APF series + (window, hop) + optional phase markers; return a dict of metrics. No I/O. Easy to unit-test.

Key adaptations
  • Reshape: x = traj.reshape(-1, 1) → [T, 1]
  • Cepstrum: CepstrumStability().compute_cepstrum(x) reused
  • Peak: compute_cepstral_peak(ceps, min_quef_idx=max(1, T//8)) override
  • Stationarity: own implementation, rolling mean over W with hop H; pass if |μ_window - μ_global| / σ_global < 1.0 for ≥ 90 % of windows
  • F1 phase + CV: imports from plan02_metrics_per_cell.py (already vetted)
  • MSC SNR: optional (single-page MSC is degenerate but still computable; off by default)

5.2 · plan03_sweep.py

What it does

CLI driver. Reads manifest, iterates cells, iterates (W, H) combos, calls kernel, streams output to CSV with per-row reproducibility hash. ~120 LOC.

CLI surface
  • --cells-dir · v2_full/cells path
  • --manifest · manifest.csv path
  • --output-csv · per-row results
  • --output-json · aggregated summary (calls aggregator)
  • --windows default {8, 16, 32, 64}
  • --hop-ratios default {0.25, 0.50, 1.00} → hop = round(W × ratio)
  • --workload-filter regex
  • --min-pairs default 4 (skip ultra-short cells)
  • --workers default 1 (multiprocessing optional)
  • --seed default 0 (for any bootstrap inside)

5.3 · plan03_aggregate.py

What it does

Reads sweep.csv. Groups by (workload, W, H). Applies gates G1-G5. Picks winner per workload. Writes recommendation.json + markdown table. Computes Pareto frontier across workloads for the joint-winner promotion clause.

Decision rule (DS approved)
  • Gates AND of (stationarity, spectral coverage, defining metric, hop ratio, n_windows floor)
  • Among passing combos, smallest window; tie-break on hop = W/2; tie-break on higher median metric
  • If no combo passes a workload → record winner=null + cite fallback path from design doc §4
  • Simpson's-paradox guard: flag if winning W differs across iv strata; require human sign-off before merging

SECTION 07

Key disputes + decisions

Dispute 1 · 1-D APF vs 2-D dump matrix

The fight

SA argued v2's streaming output (1-D APF only) is sufficient; ML wanted a small re-capture with raw dump retention so cepstrum could run on the page-level 2-D matrix (richer signal).

The resolution

DE settled it by measuring: the existing CepstrumStability.compute_cepstrum accepts shape [T, N] with N=1 as a mathematically valid degenerate case. The rfft + log-magnitude pipeline is dimension-agnostic. 1-D path wins on cost (no re-capture, ~15 s sweep). 2-D path queued as a future Plan-N upgrade if 1-D recommendation proves brittle in production use.

Dispute 2 · Window range

The fight

The design doc fixed W ∈ {32, 64, 128, 256, 512}. XD said 0/90 cells admit W=128. ML wanted {8, 16, 32, 64}. DS warned W=8 hits cepstral-noise floor.

The resolution

Compromise: W ∈ {8, 16, 32, 64}. Keep W=64 in sweep to surface its failure mode (only 5 / 90 cells compatible) for the report. Recommend W=8 only when cepstral metrics are not the primary deliverable for that workload (steady workingset CV does not require fine cepstral resolution).

Dispute 3 · Acceptance thresholds

The fight

Design doc said F1 ≥ 0.8 and CV ≤ 0.15. EE pointed out v2's measured ransom iv=500 mean F1 is exactly 0.80 — threshold sits at the population mean, trivial to pass. Same critique for CV ≤ 0.15 vs v2's 0.022 median (~7x loose).

The resolution

Tighten to v2-grounded floors: F1 ≥ 0.67 per cell AND median ≥ 0.85 (ransom); CV ≤ 0.05 at d ≤ 120 s AND CV ≤ 0.15 at d=300 s (workingset). Add v2 regression guard at the aggregate level: median F1 ≥ 0.95 AND median CV ≤ 0.05 across the chosen workload's winning combo.

Dispute 4 · sqlite OLTP scope

The fight

Design doc lists app_sqlite_oltp_v2 as a third workload. v2 only captured the 2 binaries (ransom, workingset). Adding sqlite requires fresh capture (3-5 h).

The resolution

Defer sqlite to Plan 03.5 or v3. Plan 03 ships with 2-workload scope. Caveat documented in recommendation.json. Justification: keeps the "zero new captures" commitment from the design doc's cost section.

Dispute 5 · Stationarity test source

The fight

Design doc says "use Phase 1's StabilityValidator for stationarity." That class was built for 2-D PLV/MSC on [T, N] page matrices, not 1-D APF rolling-mean stationarity.

The resolution

Ship our own stationarity_per_window inline (~15 LOC in the kernel). Test: |μ_window - μ_global| / σ_global < 1.0 for ≥ 90 % of windows. Plan 03 doc updated to clarify this is a 1-D stationarity test, not the canonical StabilityValidator.

Dispute 6 · Sub-pilot vs full sweep

The fight

SA wanted a 6-cell pilot mirroring B+3.1's validation. EE pushed for direct full sweep.

The resolution

Full sweep. Reasoning: compute is ~15 s sequential. A pilot would de-risk script bugs only, not findings. Cheaper to ship a --combos <subset> --max-cells 6 dry-run mode for script validation pre-merge, then run the full sweep in one shot.

Validation strategy

New C7 operational claim

Claim text

"A plan03_recommendation.json exists in the session output directory, validates against the v1 schema, and the per-workload winner passes gates G1-G5 when evaluated against plan03_sweep.csv."

Validator wiring

SECTION 10

Statistical design

Test framework

  • Within-cell comparison across (W, H): Wilcoxon signed-rank or Friedman with cell as block. Non-parametric because F1 is bounded [0,1] with mass at exactly 0 and 1; cepstral peak index is ordinal.
  • Stationarity / spectral-coverage rate: two-proportion z-test; Fisher's exact when any group n < 5.
  • Bootstrap CIs: BCa at 10 000 iterations, stratified resample within (iv, d) blocks. Reported as the headline metric in recommendation.md.

Sample size + power · honest accounting

Power is low. v2 effective n per (workload, iv) after replicate-median collapse: n=3. Pooled across iv: n=15 per workload. Against effect sizes of Cohen's d ≈ 0.26 (ransom F1 delta 0.10) and d ≈ 0.17 (workingset CV delta 0.05), two-sample power at α=0.05 is ~23 % (ransom pooled) and ~13 % (workingset pooled). Real claims must be ranking-based, not absolute-CI-based.

Multiple-comparisons handling

Holm-Bonferroni FWER control, per-workload separately (m=12 tests per workload, not pooled). Bonferroni is uniformly weaker; FDR is wrong here because Plan 03's single-selection decision propagates downstream (false positives have physical cost). Friedman's omnibus test gates whether pairwise tests are even meaningful.

Replicate non-independence

3 replicates per (workload, iv, dur) share VM state and time-of-day. K-W on raw 9 cells per (workload, iv) inflates the test statistic. Mitigation: collapse to median per (workload, iv, dur) → n=15 per workload at the dur level, or n=5 per (workload, iv). Honest.

Simpson's-paradox guard

If pooled winner across iv differs from any per-iv stratum winner, recommendation flags simpson_paradox_warning and the aggregator refuses to auto-pick; operator must sign off. Same pattern as Plan 02 v1/v2 split-by-workload was guarded against.

Bootstrap report format

Workload: sandbox_ransom_batched Winner: window=16, hop=8 median F1 = 1.000 [BCa 95% CI: 0.667, 1.000] (n=21 compatible cells) P(F1=1.0) = 14/21 = 0.667 (robustness check on bimodal distribution) v2 baseline: median F1 = 1.000 → delta = 0.000 Bootstrap: 10 000 iterations, BCa, stratified by (iv, d)

Gate refinements during implementation

First sweep returned winner=null for both workloads — every combo failed at least one gate. Root-cause review identified two design defects in the original gate spec.

defectcausefix
G3 ransom uses F1, but F1 is window-independent plan02_metrics_per_cell.detect_boundaries_diff takes only the trajectory, no (W, H) parameters. F1 score is per-cell, identical across all combos for the same cell. Cannot discriminate winners. Replace with median ceps_peak_snr_db ≥ 5 dB across cells with n_windows ≥ 5. SNR is window-dependent (FFT length = W) and measures how clearly the analyzer resolves the workload's rhythm at this (W, H). F1 retained for the v2 baseline regression check only.
G5 denominator includes ineligible cells Original G5 = n_cells_ge5 / total_cells ≥ 0.80. v2's d=60 cells have 5-15 snaps, cannot produce 5 windows at any sensible (W, H). They drag G5 below 0.80 unconditionally. Restrict denominator to physically-eligible cells: n_pairs ≥ W + 4H. Cells whose trajectory length cannot in principle yield 5 windows are excluded from G5 evaluation entirely. Threshold stays 0.80.

The G2 ceiling explained

Why ransom can't pass G2

G2 requires the analyzer window to cover at least twice the workload's slowest rhythm in guest time. Ransom's slowest rhythm is the encryption phase, around 20 seconds. So G2 demands W × iv ≥ 40 seconds. At iv=500 ms that needs W ≥ 80 snaps. v2's max snap count is 73 (single cell), median 22. No (W, H) in the {8, 16, 32, 64} range × any iv combination clears the 2× threshold at v2's actual snap distribution.

Why this is documented, not blocked

The Plan 03 proposal §11 already flagged this as risk R1 (snap-count feasibility ceiling). The aggregator's best-feasible fallback names G2 as the relaxed gate, so the recommendation is defensible. Raising the snap-count ceiling is Plan 05 (producer throughput). Until then, the W=8 H=4 best-feasible winner is the right operational choice — it's the smallest window with the highest cepstral SNR on the cells where it is applicable.

v2 baseline regression-guard finding

Ransom degrades_v2_baseline=true · this is a v2 ceiling, not a Plan-03 regression. The Delta-5 guard reads cell_<id>.json::analyzer_outputs.f1_phase from each v2 cell. Across the 45 ransom cells, median F1 = 0.67. The Delta-5 floor of 0.95 (set in the proposal based on the v2 §13 headline of 0.95 average) is unreachable on the per-cell distribution because v2 itself caps there. The guard fires correctly, but the cause is v2's marker-detection limitation (phase-marker alignment defect on 4-9 cells per the §08 statistical design note), not anything Plan 03 changed.

Operational implication: when downstream consumers (Plan 04 segmenter) read the recommendation, they see passes_acceptance=false for ransom and can choose to (a) use the W=8 H=4 best-feasible pick anyway, (b) wait for Plan 05 + v3 capture to lift the ceiling, or (c) operate on workingset only. The recommendation surface gives them the data needed to choose.
