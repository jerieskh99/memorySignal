D-51 · analyzer-then-delete · improvement proposal

B+3 is the right shape.
Four small deltas make it ship-ready.

Team spent a simulated week pressure-testing B+3 against the 100 GiB disk budget and the iv=100/500/2000 sweep. Top-down it is structurally sound. Bottom-up four weaknesses surfaced: synchronous +1s/snap, silent helper crash, no per-cell ack format, no completeness validator. B+3.1 fixes all four in ~135 LOC across three files. No new daemons, no new dependencies, backward compatible with the current Plan-02 orchestrator.

SECTION 01

Executive summary

What B+3 was

Producer writes 1 GiB dump per snap. After each pmemsave, a small Python helper reads the previous dump + current dump, computes how many memory pages changed (active page fraction), writes one line to a trajectory file, deletes the previous dump. Two dumps maximum live on disk at any moment. Cap on disk: ~2 GiB regardless of cell length.

What changes in B+3.1
  • Δ-1 · helper runs asynchronously (background per-pair worker) instead of blocking the producer cycle for +1 s
  • Δ-2 · explicit per-pair ack file (.apf_done) closes the silent-crash detection gap
  • Δ-3 · trajectory written as checksummed JSONL; consumer validates lineage
  • Δ-4 · post-cell completeness validator (C6) added to plan02_validate_session.py
Net effect. Same disk budget (~2-3 GiB peak). ~50 % less wall-clock cost per cell. Helper failures surface within seconds rather than at cell-end. Trajectory integrity provable on disk.

SECTION 03

Baseline B+3 review · what we started from

Intent

Enable D-51 (analyzer-then-delete metrics) for cells of any duration on a 100 GiB-free host. Original D-51 keeps all dumps until cell-end; that breaks for cells with more than ~150 GiB worth of dumps. B+3 caps disk at 2 GiB by computing APF inline as soon as the next dump lands.

Concrete spec
  • plan02_apf_helper.py · standalone Python · 75 LOC
  • Producer bash hook · 10 LOC · gated by TIMING_APF_STREAM=1
  • plan02_run.py wiring · 15 LOC · sets env-vars + reads trajectory file post-cell
  • APF computed via numpy memmap XOR; result appended to apf_trajectory.jsonl; previous dump deleted
  • Disk peak: 2 GiB (prev + curr)
  • Producer cycle latency: +1 s per snap (synchronous helper call)

SECTION 04

Top-down analysis

Objective, dependencies, success criteria, and structural risks viewed at the pipeline level.

Objective alignment

What we're trying to achieve

Populate analyzer_outputs.{f1_phase, cv_workingset, n_windows} on every cell of a v2 sub-pilot while staying under 100 GiB disk usage and keeping the operator's per-cell wall-clock within a 2× factor of v1's measured baseline (~1.5 s host_dt).

Does B+3 meet it?

Disk: yes (2 GiB peak). Wall-clock: not within 2× target. At iv=500 ms the +1 s helper adds ~67 % per cycle. At iv=100 ms it adds 800 % per cycle, pushing pause-fraction past 99 %. The latency penalty scales inversely with iv, hurting exactly the regime (fast iv) where F1 / CV resolution matters most.

Structural risks

  • R1 · latency scaling · per-cycle +1 s caps the usable iv range. Fast iv (≤ 250 ms) becomes prohibitively slow; pause-fraction rises past 95 %.
  • R2 · silent partial failure · if helper dies mid-cell, dumps stop being analyzed but producer continues writing them; disk fills silently within ~50 snaps.
  • R3 · trajectory integrity · no sequence-number check; if pairs complete out of order (cannot under sync B+3, but will under async) the trajectory becomes ambiguous.
  • R4 · debuggability · helper failures don't surface to plan02_validate_session.py's C-claims; pilot might report "all green" while half the cells have null analyzer_outputs.
SECTION 05

Bottom-up analysis

Component-by-component review with implementation details and failure modes.

5.1 · plan02_apf_helper.py

What this file does

Reads two dump files, counts differing 4 KiB pages, writes one JSON line, deletes the older dump.

Weaknesses found
  • Python startup ~50-100 ms · fresh subprocess per snap is wasteful
  • No structured exit code · producer cannot distinguish numpy-missing from io-error
  • Delete-before-flush race window · if rm succeeds but JSONL flush fails, trajectory has a gap pointing at a dump that no longer exists
  • No checksum on the dump-pair before delete · silent corruption not caught

5.2 · Producer bash hook

What the hook does

After each pmemsave, invokes python3 plan02_apf_helper.py --prev ... --curr .... Producer's main loop blocks until helper returns.

Weaknesses
  • Synchronous blocking is the dominant latency cost
  • Helper exit code not interpreted · any non-zero just logs
  • No mechanism to drain a backlog if helper is slow under host contention
  • If helper is killed by OOM, producer never knows · continues writing dumps that pile up

5.3 · plan02_run.py wiring

What wiring does

When cell.keep_dumps=True, exports TIMING_APF_STREAM=1 + TIMING_APF_JSONL=<path> before launching producer. After producer ends, the D-51 hook reads the trajectory file instead of scanning dumps.

Weaknesses
  • No completeness check · D-51 reads whatever happens to be in the JSONL · gaps not surfaced
  • If only K out of N snaps got APF, F1/CV are computed on a biased subsample · DS flagged this as significant
  • No cell-end barrier · if helper is still running when producer exits, race window between "producer stopped" and "all APF lines flushed"

SECTION 07

Key disagreements + how they resolved

Dispute 1 · per-line hashing (Day 4)

DE position

Every JSONL line should carry a 32-bit hash for integrity. Cheap, auditable, lets a reviewer prove nothing was edited.

ML pushback · adopted

Hash duplicates work that exit-code + ack file already do. Hashing every line adds ~2 µs of CPU and ~16 chars of JSONL bloat per snap × thousands of snaps = wasted bytes for a problem we already solved upstream. Compromise: hash only the final sentinel.

Dispute 2 · sync vs async helper (Day 2)

EN initial

Stay sync. Easier to reason about. Disk peak guaranteed 2 GiB. No race conditions to debug.

SA pushback · adopted

+1 s/snap kills the iv=100 ms regime. We collapse the very regime where F1 detection matters most. Async raises disk peak from 2 GiB to ~3 GiB (still trivial). Race conditions resolved by ack-file + cell-end barrier. Adopted.

Dispute 3 · scope creep (Day 3-4)

XD proposal

While we are touching the producer, also add proper huge-page-aware diff (instead of 4 KiB diff) for ~10× speedup on x86_64.

PM pushback · rejected

Out of scope. The 4 KiB diff is already fast enough under async. Huge-page diff requires kernel cooperation and changes the metric definition. Park for a future "v3" if needed.

Dispute 4 · sentinel format (Day 4)

DE proposed JSON

{"final": true, "n_pairs": N, "chk": "fnv1a-of-all-apf-values"}

EE simplified · adopted

Drop the chk hash (already debated in Dispute 1). Sentinel becomes just {"final": true, "n_pairs": N, "n_ok": K, "n_failed": M}. Validator reads three integers, checks K + M = N - 1, fails C6 if not.

SECTION 08

Improved plan · B+3.1 specification

Four deltas over B+3. Each delta is small, additive, and gated by an env-var so the existing flow is untouched when nothing is set. Below each delta has three parts: what it is, why we need it, and how it is computed / implemented.

What is a "delta"? A delta (Δ) is a self-contained change from baseline. Each delta resolves one specific weakness surfaced during the week-long review (Sections 04-06). Together they form B+3.1, the production-ready version. Numbered Δ-1 through Δ-4 in the order the team converged on them, not in implementation order.

Δ-1 · async helper + cell-end barrier

What it is

The APF-computing helper (which reads two dumps and counts how many memory pages changed between them) now runs in the background instead of making the producer wait. After each pmemsave, the producer launches the helper as a separate process and continues immediately to the next snapshot. At the end of the cell, the producer waits ("barriers") for all background helpers to finish before declaring the cell done.

Why we need it

In B+3 (the baseline), the producer blocks for ~1 second per snap while the helper computes APF on a 1 GiB dump. At intervalMsec = 100 ms the producer's natural cycle is ~125 ms; adding +1 s/snap pushes the cycle to ~1.1 s, which is 9× slower. Pause-fraction climbs past 95 % and the iv=100 regime — the one where F1 detection of fast workload phases matters most — becomes prohibitively expensive.

Async hides the helper's compute behind the producer's natural waiting period (suspend → pmemsave → resume → sleep). The +1 s/snap penalty drops to roughly zero because the helper finishes during the next snap's normal cycle.

How it works in practice
  • Producer writes curr.raw via pmemsave (as before).
  • Producer immediately does python3 plan02_apf_helper.py ... & — the trailing & backgrounds the process. Producer continues to virsh resume + the next sleep cycle.
  • The background helper reads prev.raw and curr.raw, computes APF (~1 s), appends one line to apf_trajectory.jsonl, deletes prev.raw, writes a per-pair .apf_done ack file, exits.
  • Producer keeps a count of spawned helpers. When the cell duration elapses, the producer enters the cell-end barrier: it polls the directory for ack files, waiting until n_ack == n_pairs_expected OR a 30 s timeout expires.
  • Barrier complete → producer writes the final sentinel line and exits.
Computation cost

APF computation per pair is unchanged: numpy.memmap XOR over two 1 GiB files → reshape to [n_pages, 4096] → any(axis=1).sum() / n_pages. ~1 s wall-clock on the host's SSD.

Async amortizes this: while helper-i runs in the background, the producer is doing pmemsave-(i+1) which also takes ~1 s. The two phases overlap. Effective per-cycle cost drops from ~1.5 s + 1 s = 2.5 s (sync) to ~1.5 s (async).

PRODUCER CYCLE · B+3.1 DISK STATE ─────────────────────── ────────────── pmemsave → curr.raw prev + curr (2 GiB) ↓ spawn apf_helper(prev, curr) [NON-BLOCKING] prev + curr + helper-in-flight ↓ (3 GiB transient) virsh resume + sleep intervalMsec ↓ next cycle: pmemsave overlaps with prior helper (cleared to 2 GiB once helper rm-s prev) ↓ ... cell continues ... ↓ producer ends · BARRIER: wait until all ack files 2 GiB → 1 GiB → 0 written OR 30 s timeout ↓ producer emits final sentinel with n_ok, n_failed ↓ D-51 reads trajectory · computes F1/CV

Δ-2 · ack file + structured exit codes

What it is

Every helper invocation writes a tiny <curr_dump_basename>.apf_done JSON file alongside the dump directory. The file contains the sequence number, prev/curr paths, exit code, helper duration, and the APF value. The helper also returns a structured exit code (0 = ok, 1 = transient io, 2 = numpy missing, 3 = disk full, 4 = corrupt dump) instead of just success/fail.

Why we need it

In B+3, if the helper dies mid-cell (OOM kill, segfault, disk full, corrupt dump), the producer never finds out. The producer keeps writing snapshots; the helper backlog grows; disk fills silently. The cell completes, the trajectory file has gaps, but no signal anywhere in the pipeline indicates "things went wrong".

Ack files give the producer a real-time signal that each pair was processed. Structured exit codes let the producer react appropriately: transient errors get logged and continue; numpy missing aborts the cell; disk full triggers backpressure.

How it works

The helper writes its ack file as the LAST action before exiting. The ack file IS the proof that the pair completed cleanly. Producer's barrier counts ack files, not just helper PIDs.

If the helper crashes before writing the ack, no ack file appears — the barrier detects the missing one and records it in the final sentinel as a gap.

Ack file format 
{
  "seq": 12,
  "prev": "memory_dump-20260524000012.raw",
  "curr": "memory_dump-20260524000013.raw",
  "exit_code": 0,
  "helper_duration_ms": 743,
  "apf": 0.1342
}

Writing the ack file is atomic from the producer's perspective: either the file exists (success signal) or it does not (failure signal). No half-states.

codemeaningproducer reaction

1transient io errorlog · continue · gap recorded

3disk fulltrigger Day-9 disk-preflight equivalent · fail cell

Δ-3 · seq-numbered JSONL + final sentinel

What it is

Each line in the trajectory file carries an explicit seq field — an integer counting from 0 — so the reader can detect missing pairs. When the producer's barrier finishes, the producer appends one last line called the final sentinel: a JSON object with n_pairs_expected, n_ok, n_failed, and the list of gap_seqs. The sentinel is the trajectory file's "I am complete and self-describing" marker.

Why we need it

Under async (Δ-1), helpers can finish out of order. Helper-7 might write its line before helper-5 if helper-5 was slow. Without seq numbers, the trajectory file is ambiguous — the reader cannot tell which pair came first or which are missing.

Without a sentinel, a partially-completed cell (producer crashed before writing it) looks identical on disk to a fully-completed one. The reader has no way to tell. Sentinel makes completeness provable by inspection.

How the reader uses these
  • Step 1: read every line of apf_trajectory.jsonl. Parse each as JSON.
  • Step 2: find the final-sentinel line (the one with "final": true). If absent, the file is incomplete — mark cell as C6 fail.
  • Step 3: sort the remaining lines by seq. Check that every integer from 0 to n_pairs_expected - 1 appears exactly once. Missing seqs go in the gap list.
  • Step 4: compute n_ok / n_pairs_expected. If < 0.95, mark cell as C6 fail (too many gaps for reliable F1/CV).
  • Step 5: downstream metrics (F1, CV) are computed only on the sorted ok lines.
Final sentinel format 
{
  "final": true,
  "n_pairs_expected": 147,
  "n_ok": 145,
  "n_failed": 2,
  "gap_seqs": [73, 91]
}

Three integers + one short list is enough to validate completeness in O(n) time. Total sentinel size: ~80 bytes regardless of cell length.

Δ-4 · new validator claim C6

What it is

A new "claim" in plan02_validate_session.py — the post-pilot health-check tool. Existing claims C1-C5 cover orchestration ("workload ran", "no settle retries", etc.). C6 covers trajectory completeness: this cell's APF trajectory exists, has a final sentinel, and at least 95 % of expected pairs landed.

Why we need it

Without C6, B+3.1 has no acceptance gate that catches "the helper silently failed for half the snaps". A cell could pass C1-C5 ("workload ran", "snap count fine", "no lock retries", "producer log clean") while having an analyzer trajectory missing 30 % of its pairs. F1 / CV computed on such trajectory would be biased.

C6 promotes "trajectory integrity" from an implicit assumption to an explicit acceptance criterion. Cells that fail C6 are excluded from the v2 recommendation aggregation.

How C6 is computed
  • Open <cell_workdir>/apf_trajectory.jsonl. If missing → C6 FAIL.
  • Read all lines. Find one with "final": true. If missing → C6 FAIL.
  • Parse n_pairs_expected and n_ok from sentinel.
  • Compute ratio = n_ok / n_pairs_expected. If ratio < 0.95 → C6 FAIL.
  • All checks pass → C6 PASS. Cell contributes to v2 aggregation.
Where C6 hooks in

In plan02_validate_session.py::evaluate_cell(), add ~25 LOC alongside the existing C1-C5 checks. C6 is operational (not informational), so a C6 failure flips the cell's overall verdict to FAIL.

For v1-era cells (no trajectory file present because the cell ran without --keep-dumps), C6 is reported as "not applicable" rather than failing. Backward compatible with v1 data.

Two C6 failures · root-cause

What happened (plain English)

Two short-duration ransom cells (60 seconds each) had the orchestrator's wait barrier expire before the last APF helper finished. The helper itself did not crash. The producer wrote 8 and 14 dumps respectively, but the last diff (pair N-1) had not yet been computed when the barrier timed out. The trajectory ends with a sentinel marking the gap honestly. Because expected pair count is so small at d=60 (just 7 and 13 total pairs), losing one drops the ratio below the 95 % threshold even though the rest of the data is intact.

Mechanism + evidence
  • cells/work/73e42bc7c685/apf_trajectory.jsonl:7 -- sentinel: {"final":true, "n_pairs_expected":7, "n_ok":6, "gap_seqs":[6], "timed_out":true}
  • cells/work/73e42bc7c685/apf_helper.log -- 0 bytes (no exception path -> no helper crash)
  • Same pattern for e61a27074f69: 12/13 pairs, last seq missing, timed_out:true
  • 17 cells in total ran with timed_out:true; 15 of them still cleared the 95 % bar because their expected pair count was high enough that one missing pair fell within tolerance (e.g. 55/56 = 98.2 %)
  • Both failing cells share workload=ransom_batched, duration_s=60 -- the lowest pair-count cohort in the matrix
Latent bug O (file under "trailing-helper barrier race"). At 1000-cell scale, an estimated ~22 cells per run will tail-miss with the current barrier deadline. Fix: extend barrier by 1x interval_ms beyond last expected snap, or wait for explicit ack of the last seq. Not a blocker for this batch -- 88/90 is acceptable for thesis -- but document for future scaling.

Bug O fix · landed and verified (commit 30de0e5)

What the fix does

Replaces the flat 30-second wall-clock barrier in wait_for_apf_helpers with a progressive idle timeout. Barrier exits when (1) all expected acks arrive, (2) no new ack appears in 15 seconds (idle quit), or (3) a 180-second hard cap fires (safety stop). Healthy cells finish quickly via the success path; only genuinely stuck helpers trigger the idle quit; only pathological hangs hit the cap. Backward compatible -- legacy callers passing only timeout_s still get the original semantics.

SECTION 14

Thesis-quality conclusions

Signature-level findings (90 cells, 2 workload classes, 5 iv settings)

In one paragraph

The phasic-vs-steady taxonomy motivating this thesis is empirically supported at the memory-capture signature level: across 90 controlled cells, the ransomware workload produces an active-page-fraction trajectory whose coefficient of variation is 28-140x larger than the steady working-set workload's, and the gap is robust to sampling interval. The optimal sampling interval is workload-dependent and non-monotonic for the phasic class: iv=500 ms produces both saturated F1 (median 1.0) and the strongest CV signal (median 3.90), while sub-Nyquist intervals (iv=100 ms) fail through boundary-localization smearing rather than information loss. For steady workloads, iv is a near-free parameter on signature grounds; iv=250 ms wins on the joint criterion of tightest CV and lowest disk pressure under the B+3.1 helper regime. Downstream ML should train iv-stratified features rather than a single global iv, because the phasic-class signature is brittle to fine sampling in a way the steady-class is not -- a finding that argues for adaptive-iv capture in any production-facing extension of this pipeline.

Defensible numerical claims
  • Ransom CV median = 2.81 (range 0.09 - 6.79). Workingset CV median = 0.037 (range 0.009 - 1.26). Ratio: 76x at median.
  • Ransom F1 vs iv: Kruskal-Wallis H=14.61, p=0.0056; effect localized to iv=100 collapse.
  • Workingset CV vs iv: K-W H=2.48, p=0.648; not significant at n=9 per group.
  • Recommended iv: ransom 500 ms, workingset 250 ms.
  • B+3.1 implementation: 88/90 operational PASS, 0 helper crashes, 0 disk-full events, persistent footprint 4.7 MiB.

Why iv=100 fails for the phasic class (boundary-localization smearing, not aliasing)

What the data shows

At iv=100 ms with abundant snaps (>= 20 per cell, 4 of 9 cells), ransom F1 is bimodal and never reaches 1.0; median is 0.67. Compare iv=250 ms with the same snap budget: 0/6 F1=0, perfect detection. So the failure is not snap starvation. Inspecting the raw APF trajectories of failing cells, two patterns emerge: (a) the encryption-start transient occupies multiple 100 ms windows, so the APF series stays high-and-flat with no single pair carrying the boundary "jump" the detector keys on; (b) when the trajectory is impulsive, the spike argmax drifts off the truth index by 4-7 pairs, producing FP + FN. By iv=500 ms the very first pair captures the encryption-start cleanly and F1 saturates.

Implication for thesis chapter

Frame this as a temporal-resolution mismatch, not aliasing: the sampling window must be wider than the workload's effective phase-boundary duration, otherwise the boundary is fragmented across pairs rather than missed. Argues against the naive "finer iv = better resolution" heuristic. Also argues for two-stage capture in future work: coarse iv to localize boundaries, then targeted fine iv inside identified phase windows.

What v2 confirms about v1

  • Direction correct, magnitude refined. v1's iv=250 / iv=1000 split-by-workload was directionally right (do not over-sample steady; do not under-sample phasic). v2 measured F1/CV refined the exact iv per workload.
  • Producer overhead argument from v1 was conservative. v1 picked iv=1000 for workingset on disk-pressure grounds; v2 with B+3.1 streaming proves iv=250 is feasible (4.7 MiB persistent footprint).
  • iv=100 cliff is reproducible. Both v1 sub-pilot (Day-14 evidence) and the full v2 run show ransom F1 collapses at iv=100. Now characterized mechanistically.

Open questions for follow-up work

  1. Phase-marker alignment. 4 cells with high snaps but F1=0 (two ransom iv=500/1000) likely indicate ground-truth phase markers misaligned with VM snapshot timestamps. Audit the SSH-side phase emission vs host-side snap timestamping before publishing F1 numbers in a thesis chapter.
  2. Duration interaction with CV. Workingset CV blows up at duration = 300 s under iv >= 1000 ms (the 256 MiB working set drifts within the observation window). Either constrain conclusions to duration <= 120 s or replicate at d=300 with larger n.
  3. Bug O (trailing-helper barrier race). 17/90 cells flagged timed_out=true; 2 fell below 95 %. Extend barrier by one iv beyond last expected snap, or wait-for-last-seq-ack. Estimated ~10 LOC fix in plan02_run.py::wait_for_apf_helpers.
  4. Disk transient over-budget (59.5 GiB swing). Per-cell quota enforcement needed before scaling to 1000-cell or multi-host topologies.
  5. n=3 per (workload, iv, duration) is power-thin. Workingset CV K-W is non-significant (p=0.648). A power calculation against observed within-group std suggests ~20 replicates needed for an 0.05 median difference at 80 % power.
