D-51 · analyzer-then-delete · improvement proposal

B+3 is the right shape.
Four small deltas make it ship-ready.

Team spent a simulated week pressure-testing B+3 against the 100 GiB disk budget and the iv=100/500/2000 sweep. Top-down it is structurally sound. Bottom-up four weaknesses surfaced: synchronous +1s/snap, silent helper crash, no per-cell ack format, no completeness validator. B+3.1 fixes all four in ~135 LOC across three files. No new daemons, no new dependencies, backward compatible with the current Plan-02 orchestrator.

SECTION 01

Executive summary

What B+3 was

Producer writes 1 GiB dump per snap. After each pmemsave, a small Python helper reads the previous dump + current dump, computes how many memory pages changed (active page fraction), writes one line to a trajectory file, deletes the previous dump. Two dumps maximum live on disk at any moment. Cap on disk: ~2 GiB regardless of cell length.

What changes in B+3.1
  • Δ-1 · helper runs asynchronously (background per-pair worker) instead of blocking the producer cycle for +1 s
  • Δ-2 · explicit per-pair ack file (.apf_done) closes the silent-crash detection gap
  • Δ-3 · trajectory written as checksummed JSONL; consumer validates lineage
  • Δ-4 · post-cell completeness validator (C6) added to plan02_validate_session.py
Net effect. Same disk budget (~2-3 GiB peak). ~50 % less wall-clock cost per cell. Helper failures surface within seconds rather than at cell-end. Trajectory integrity provable on disk.

SECTION 03

Baseline B+3 review · what we started from

Intent

Enable D-51 (analyzer-then-delete metrics) for cells of any duration on a 100 GiB-free host. Original D-51 keeps all dumps until cell-end; that breaks for cells with more than ~150 GiB worth of dumps. B+3 caps disk at 2 GiB by computing APF inline as soon as the next dump lands.

Concrete spec
  • plan02_apf_helper.py · standalone Python · 75 LOC
  • Producer bash hook · 10 LOC · gated by TIMING_APF_STREAM=1
  • plan02_run.py wiring · 15 LOC · sets env-vars + reads trajectory file post-cell
  • APF computed via numpy memmap XOR; result appended to apf_trajectory.jsonl; previous dump deleted
  • Disk peak: 2 GiB (prev + curr)
  • Producer cycle latency: +1 s per snap (synchronous helper call)
PRODUCER CYCLE · B+3 BASELINE DISK STATE ───────────────────────────────────── ────────────────── virsh suspend prev.raw (1 GiB) ↓ pmemsave → curr.raw prev.raw + curr.raw (2 GiB) ↓ apf_helper(prev, curr) [BLOCKING +1s] prev.raw + curr.raw (2 GiB) ↓ helper writes apf line · deletes prev curr.raw (1 GiB) ↓ virsh resume ↓ sleep intervalMsec ↓ next pmemsave makes prev=curr; curr=new next cycle
SECTION 04

Top-down analysis

Objective, dependencies, success criteria, and structural risks viewed at the pipeline level.

Objective alignment

What we're trying to achieve

Populate analyzer_outputs.{f1_phase, cv_workingset, n_windows} on every cell of a v2 sub-pilot while staying under 100 GiB disk usage and keeping the operator's per-cell wall-clock within a 2× factor of v1's measured baseline (~1.5 s host_dt).

Does B+3 meet it?

Disk: yes (2 GiB peak). Wall-clock: not within 2× target. At iv=500 ms the +1 s helper adds ~67 % per cycle. At iv=100 ms it adds 800 % per cycle, pushing pause-fraction past 99 %. The latency penalty scales inversely with iv, hurting exactly the regime (fast iv) where F1 / CV resolution matters most.

Success criteria · what 'done' looks like

Structural risks

  • R1 · latency scaling · per-cycle +1 s caps the usable iv range. Fast iv (≤ 250 ms) becomes prohibitively slow; pause-fraction rises past 95 %.
  • R2 · silent partial failure · if helper dies mid-cell, dumps stop being analyzed but producer continues writing them; disk fills silently within ~50 snaps.
  • R3 · trajectory integrity · no sequence-number check; if pairs complete out of order (cannot under sync B+3, but will under async) the trajectory becomes ambiguous.
  • R4 · debuggability · helper failures don't surface to plan02_validate_session.py's C-claims; pilot might report "all green" while half the cells have null analyzer_outputs.
SECTION 05

Bottom-up analysis

Component-by-component review with implementation details and failure modes.

5.1 · plan02_apf_helper.py

What this file does

Reads two dump files, counts differing 4 KiB pages, writes one JSON line, deletes the older dump.

Weaknesses found
  • Python startup ~50-100 ms · fresh subprocess per snap is wasteful
  • No structured exit code · producer cannot distinguish numpy-missing from io-error
  • Delete-before-flush race window · if rm succeeds but JSONL flush fails, trajectory has a gap pointing at a dump that no longer exists
  • No checksum on the dump-pair before delete · silent corruption not caught

5.2 · Producer bash hook

What the hook does

After each pmemsave, invokes python3 plan02_apf_helper.py --prev ... --curr .... Producer's main loop blocks until helper returns.

Weaknesses
  • Synchronous blocking is the dominant latency cost
  • Helper exit code not interpreted · any non-zero just logs
  • No mechanism to drain a backlog if helper is slow under host contention
  • If helper is killed by OOM, producer never knows · continues writing dumps that pile up

5.3 · plan02_run.py wiring

What wiring does

When cell.keep_dumps=True, exports TIMING_APF_STREAM=1 + TIMING_APF_JSONL=<path> before launching producer. After producer ends, the D-51 hook reads the trajectory file instead of scanning dumps.

Weaknesses
  • No completeness check · D-51 reads whatever happens to be in the JSONL · gaps not surfaced
  • If only K out of N snaps got APF, F1/CV are computed on a biased subsample · DS flagged this as significant
  • No cell-end barrier · if helper is still running when producer exits, race window between "producer stopped" and "all APF lines flushed"

5.4 · apf_trajectory.jsonl format

fieldtypestrengthweakness

SECTION 06

Week-long team audit · daily meetings

Five working days. PM ran a standing meeting each morning. Decisions logged inline.

Day 1 Kickoff · stake out the scope PM · 60 min
  • PM defined scope: review B+3 against 100 GiB free disk + reasonable wall-clock + thesis-grade reproducibility.
  • SA: structurally B+3 is correct; latency is the headline risk.
  • ML: trajectory completeness is the unspoken assumption · need a guard.
  • EN: helper exit codes are too coarse.
  • XD: cell-level wall-clock changes vs v1 must be recorded or future analysis confounds.
  • DE: trajectory needs a checksum or sequence-validation layer.
  • DS: out-of-order pair completion (under any async variant) needs explicit handling.
  • EE: validator does not yet check this; new C6 claim needed.
  • Decision D-1: proceed with B+3 as the structural baseline. Each agent owns one improvement vector by Day 2.
Day 2 Latency model · async or stay sync? SA + EN · 90 min
  • SA proposed: fork helper as background subprocess immediately after pmemsave returns; producer continues to next cycle without waiting. Cell-end barrier joins all helpers.
  • EN concern: async invites two dumps + one in-flight = 3 GiB peak; still under budget. Acceptable.
  • EN concern #2: how do we know all helpers finished before D-51 reads the trajectory? Need explicit join.
  • ML: out-of-order completion now possible. Need seq-numbered JSONL lines + post-cell sort.
  • DS: out-of-order has no statistical impact after sort because APF is order-independent per pair. F1 detector reads sorted trajectory.
  • Decision D-2: adopt async helper. Cell-end barrier required. Trajectory sorted by seq before D-51 reads.
Day 3 Resilience · ack files + structured exit codes EN + DE · 75 min
  • EN proposed exit-code taxonomy: 0=ok, 1=transient-io, 2=numpy-missing, 3=disk-full, 4=corrupt-dump.
  • Producer reaction: 0=continue; 1=log+continue; 2=abort cell; 3=enter backpressure; 4=mark pair as gap.
  • DE proposed: each helper writes <curr_dump>.apf_done ack file before deleting prev. Producer counts ack files; if backlog grows past N, pause new pmemsave (backpressure).
  • SA agrees but caps backpressure at simple "wait up to 5 s for helpers to drain" before forcing the next cycle (avoids hang).
  • EN: ack file content = JSON with seq, prev path, curr path, exit_code, helper_duration_ms. Doubles as audit trail.
  • Decision D-3: ack file per pair · exit-code taxonomy fixed · producer reads acks at cell-end and reports ack_count, expected_count, gap_seq[] in the producer.log.
Day 4 Trajectory integrity · checksum + completeness validator DE + ML + EE · 90 min
  • DE proposed: helper computes a 32-bit fnv1a hash of (prev_dump_sha1, curr_dump_sha1, apf_float64_bytes) per line. JSONL gets a "chk" field. Reader can verify nothing was edited or truncated mid-line.
  • ML: hash is overkill if exit-code + ack-file already cover the failure modes. Push back.
  • EE compromise: keep the hash only on the LAST line of the JSONL as a checkpoint, plus seq-number gap detection. Full per-line hashing not needed.
  • DE accepts. JSONL ends with a final {"final": true, "n_pairs": N, "chk": "..."} sentinel.
  • EE drafted new C6 claim: apf_trajectory.jsonl exists AND has 'final' sentinel AND n_pairs equals (n_snaps - 1). Add to plan02_validate_session.py.
  • Decision D-4: seq-numbered JSONL + final sentinel + C6 validator claim · no per-line hashing.
Day 5 Convergence · final spec frozen PM · 60 min
  • PM read back four agreed deltas:
  •   Δ-1 · async helper + cell-end barrier (SA + EN)
  •   Δ-2 · ack-file per pair + structured exit codes (EN + DE)
  •   Δ-3 · seq-numbered JSONL + final sentinel (DE + ML)
  •   Δ-4 · C6 completeness claim in validator (EE)
  • XD reviewed: cell-level host_dt expected to change vs v1 by ~10-20 % (async overhead is much lower than sync). Need to record async-mode flag in cell metadata so future analyses know.
  • DS reviewed: with sort-by-seq + sentinel check, statistical interpretation matches the sync model. No mathematical compromise.
  • Decision D-5: B+3.1 frozen. Implementation cost estimate: ~135 LOC across 3 files + ~30 LOC tests. No new dependencies. Backward compatible.
  • PM: ship after operator approves.
SECTION 07

Key disagreements + how they resolved

Dispute 1 · per-line hashing (Day 4)

DE position

Every JSONL line should carry a 32-bit hash for integrity. Cheap, auditable, lets a reviewer prove nothing was edited.

ML pushback · adopted

Hash duplicates work that exit-code + ack file already do. Hashing every line adds ~2 µs of CPU and ~16 chars of JSONL bloat per snap × thousands of snaps = wasted bytes for a problem we already solved upstream. Compromise: hash only the final sentinel.

Dispute 2 · sync vs async helper (Day 2)

EN initial

Stay sync. Easier to reason about. Disk peak guaranteed 2 GiB. No race conditions to debug.

SA pushback · adopted

+1 s/snap kills the iv=100 ms regime. We collapse the very regime where F1 detection matters most. Async raises disk peak from 2 GiB to ~3 GiB (still trivial). Race conditions resolved by ack-file + cell-end barrier. Adopted.

Dispute 3 · scope creep (Day 3-4)

XD proposal

While we are touching the producer, also add proper huge-page-aware diff (instead of 4 KiB diff) for ~10× speedup on x86_64.

PM pushback · rejected

Out of scope. The 4 KiB diff is already fast enough under async. Huge-page diff requires kernel cooperation and changes the metric definition. Park for a future "v3" if needed.

Dispute 4 · sentinel format (Day 4)

DE proposed JSON

{"final": true, "n_pairs": N, "chk": "fnv1a-of-all-apf-values"}

EE simplified · adopted

Drop the chk hash (already debated in Dispute 1). Sentinel becomes just {"final": true, "n_pairs": N, "n_ok": K, "n_failed": M}. Validator reads three integers, checks K + M = N - 1, fails C6 if not.

SECTION 08

Improved plan · B+3.1 specification

Four deltas over B+3. Backward compatible: when TIMING_APF_STREAM is absent, producer behaves identically to today.

Δ-1 · async helper + cell-end barrier

PRODUCER CYCLE · B+3.1 DISK STATE ─────────────────────── ────────────── pmemsave → curr.raw prev + curr (2 GiB) ↓ spawn apf_helper(prev, curr) [NON-BLOCKING] prev + curr + helper-in-flight ↓ (3 GiB transient) virsh resume + sleep intervalMsec ↓ next cycle: pmemsave overlaps with prior helper (cleared to 2 GiB once helper rm-s prev) ↓ ... cell continues ... ↓ producer ends · BARRIER: wait until all ack files 2 GiB → 1 GiB → 0 written OR 30 s timeout ↓ producer emits final sentinel with n_ok, n_failed ↓ D-51 reads trajectory · computes F1/CV

Δ-2 · ack file + structured exit codes

Each helper invocation writes <curr_dump_basename>.apf_done in the same directory before exiting.

{
  "seq": 12,
  "prev": "memory_dump-20260524000012.raw",
  "curr": "memory_dump-20260524000013.raw",
  "exit_code": 0,
  "helper_duration_ms": 743,
  "apf": 0.1342
}

Exit codes:

codemeaningproducer reaction
0ok · APF written · prev deletedcontinue
1transient io errorlog · continue · gap recorded
2numpy missing / import failureabort cell after 3 consecutive code-2s
3disk fulltrigger Day-9 disk-preflight equivalent · fail cell
4dump file corrupt or zero-bytemark pair as gap · continue

Δ-3 · seq-numbered JSONL + final sentinel

Each apf line carries an explicit seq. After producer ends and barrier completes, producer appends:

{"final": true, "n_pairs_expected": 147, "n_ok": 145, "n_failed": 2, "gap_seqs": [73, 91]}

Reader (D-51 + validator) sorts by seq, checks for gaps, refuses to compute F1/CV if more than 5 % gaps.

Δ-4 · new validator claim C6

Add to plan02_validate_session.py:

  • C6 · apf_trajectory completeness · file exists · contains final sentinel · n_ok / n_pairs_expected ≥ 0.95
  • Per-cell: cell_workdir/apf_trajectory.jsonl · sentinel parsed · pass/fail recorded
  • C6 is operational · failures gate the cell from contributing to v2 recommendation

Acceptance gates

  • All 6 cells pass C6 (new completeness gate)

  • Peak disk during any cell < 5 GiB (observed via heartbeat)
