VI Why long traces confused

Three mechanisms, one bug.

Concrete signal-processing pathologies explain the high-confusion classifier results on long traces that should have been easy to separate.

i

Window-duration collapse

With Δt ≈ 6.6 ms, a 128-frame window covers 0.85 s of guest time — not 12.8 s. Any rhythm slower than ~1 s of guest activity has its identity averaged into the baseline.

→ sqlite checkpoint, sandbox phases, slowburn all collapse to "write activity over 0.85 s".
ii

Cepstral peak mis-mapping

A peak at frame k was read as period k × 100 ms. Actual spacing is 6.6 ms, so the peak is k × 6.6 ms — 15× shorter. Expected k = 10 sits at k ≈ 151.

→ even when a rhythm is present it lands in the wrong bin; classifier never sees it.
iii

Non-stationary Δt

Suspend latency drifted 0.97 → 4.20 → 5.80 s across three snapshots. Cepstrum, MSC, PLV all assume stationary sampling. Drift smears peaks across many bins.

→ steady workloads look multi-modal; multi-modal ones look chaotic.
iv

Pause-fraction noise floor

The VM is paused 99.8 % of host wall-clock. Anything host-driven (NTP, host disk cycles, neighbouring VMs) leaks into the captured state in a workload-correlated way — same host context for every class.

→ shared noise floor squeezes the discriminative margin.
VII Recommended next moves

Don't tune anything else until this lands.

All downstream tuning experiments are gated on this fix. Re-collecting the dataset is non-negotiable.

1
Immediate · code

Fix the sleep

  1. On pcrserral run command -v bc. Empty → apt install bc.
  2. Independently, rewrite the producer sleep bc-free: sleep 0.$(printf "%03d" $iv) with a ≥1000 branch, or read -t $((iv))e-3.
  3. Add a startup self-check: 10-snapshot sanity loop that aborts if guest_dt_mean deviates from intervalMsec by > 10 %.
2
Then · validate

Re-run Experiment 1

  1. --duration 300 --test-command bash ~/VM_executables_phase2/scripts/run_phase2_min.sh. Confirm guest_dt_mean ≈ intervalMsec ± 5 %.
  2. Characterise the suspend-latency drift (0.97 → 4.20 → 5.80 s). Three points was not enough to know whether it saturates.
  3. Re-collect one representative workload pair that confused before.
3
Finally · decide

Re-run the classifier

  1. If confusion collapses → timing bug was load-bearing. Proceed to plan 02 (interval tuning).
  2. If confusion persists → window/hop is the next suspect. Run plan 03 against fresh data.
  3. Until then treat every prior result from this host as suspect.
Every prior confusion-matrix from this host is suspect until at least one re-collected workload produces guest_dt_mean ≈ intervalMsec. Tuning on bad timing means tuning toward the wrong optimum.
