Plan 03 features · Plan 02 v3 dataset · classification phase

The phasic-vs-steady taxonomy
holds at the classifier level.

Plan 02 v3 produced 132 lossless cells across eleven Phase-2 workloads, with four replicates per workload-and-duration tuple. Plan 03 picked a single analyzer window of 8 samples and a hop of 4 across every workload tested. With those features in hand, a standard ensemble of classifiers, trained under leave-one-replicate-out cross-validation, asks two questions: whether the family of a held-out cell (phasic or steady) can be recovered, and whether the specific workload within that family can be identified. The answer to the first question is essentially perfect, and the answer to the second question, while modest in absolute accuracy, is interesting in exactly the way the taxonomy claim predicts.

SECTION 01

What this is

In plain words

The thesis argument is that memory-signature features can distinguish phasic workloads (encryption bursts, file scans, paced ransomware) from steady workloads (sustained memory access, working-set churn, application probes). This document tests that argument empirically. The captured trajectories from Plan 02 v3 are reduced to per-cell feature vectors using the analyzer settings that Plan 03 recommends, and three off-the-shelf classifiers attempt to recover the family label and the specific workload label from those features. The held-out evaluation uses replicate boundaries as the split, which means each fold trains on three of the four replicates and tests on the fourth.

In one sentence

A three-classifier ensemble (kNN with five neighbors, a 300-tree Random Forest, and a two-layer MLP) is trained under GroupKFold with the replicate index as the grouping variable, on 118 cells whose Plan 03 sweep produced valid features at W=8 and H=4, reporting per-class precision, recall, and F1 for the eleven-way task and for the binary phasic-versus-steady task.

Where this document sits in the project. Plan 02 produced the captures. Plan 03 selected the analyzer window and hop and computed the per-cell features. This document consumes those features and produces the classification result. It is the empirical anchor for the thesis taxonomy claim. The next layer of analyzer work, the change-point segmenter, is owned by Plan 04 and lives in plan04_segmenter_proposal.html.

Two of the ten features, cv_workingset and f1_phase, are populated only for one of the two workload families because the Plan 03 kernel dispatches its defining-metric computation by family. To make these features usable by a classifier that does not know the family in advance, the pipeline applies median imputation (from sklearn.impute.SimpleImputer) before standardisation. The imputed values are not meaningful in isolation, but they provide a sensible default that does not bias the classifier toward either family.

SECTION 05

Cross-validation scheme

In plain words

The dataset contains four independent captures (replicates) for each combination of workload and duration. The natural way to evaluate a classifier on this data is to hold out one entire replicate at a time, training on the other three. That way the model is always tested on cells it has not seen, and the held-out cells span every workload and every duration. The chosen scheme, called GroupKFold with the replicate index as the group variable, implements exactly this protocol.

In one sentence

The split uses sklearn.model_selection.GroupKFold(n_splits=4) with the replicate index as the group key, so each of the four folds trains on approximately 88 cells and tests on approximately 30, with every workload class represented in both partitions.

Two alternative schemes are weaker for this evaluation. A plain stratified k-fold would mix replicates within both partitions and inflate accuracy because the test set would share replicate-level idiosyncrasies with the training set. A leave-one-cell-out scheme would have the same problem at finer granularity. Leave-one-replicate-out is the strictest defensible split that this dataset supports without dropping the n=4 sample count below the point at which any per-class metric becomes meaningful.

SECTION 06

Models tested

Three off-the-shelf classifiers are evaluated. The intent is to read the result as "a standard ensemble can solve this task," not "this particular hyperparameter setting is optimal." Each classifier is wrapped in a scikit-learn pipeline that first imputes missing values, then standardises the features to zero mean and unit variance, and finally fits the model.

modelparametersrationale
k-nearest neighborsk = 5, distance-weighted votingA non-parametric baseline that uses local feature-space density.

multi-layer perceptronhidden layers of 64 and 32 units, ReLU activations, up to 2000 epochsA small neural network to verify that a non-linear decision surface does not dramatically outperform the simpler models.

SECTION 09

Interpretation

Why the binary task is essentially perfect

The Plan 03 features differ systematically between the two families, and the difference is large. The cepstral signal-to-noise ratio is between 4.7 and 6.4 decibels for the phasic workloads and below 3 decibels for the steady workloads. The coefficient of variation, when populated, sits between 0.06 and 0.33 for steady workloads. The active-page-fraction mean is high and constant for steady workloads and bursty for phasic workloads. These differences compound across ten dimensions, and any classifier that respects feature scaling and tree-style decision boundaries can recover them. The 100 percent figure should not be interpreted as a claim about all possible workloads: it is a claim about these eleven, on this dataset, at this analyzer setting. But the gap between the families is large enough that the result is unlikely to be a small-sample artefact.

Why the eleven-way task is harder

Within each family, the workloads share enough structure that a simple feature ensemble cannot reliably distinguish them. The four ransom variants are largely confused with one another, as are the six memory probes. The structure of these confusions is informative. The memory probes mmap_traversal and workingset_sweep share a working-set-sweep access pattern and are mutually confused; the ransom variants seq and slowburn share a per-file processing structure and are also mutually confused. The scanner achieves the highest per-class F1 at 0.60 because its multi-pass metadata access produces a distinctive cepstral peak that no other workload exhibits.

What the confusion-matrix block structure says

The fact that every error stays within the family block is the structural form of the taxonomy claim. The classifier has not been told which workloads are phasic and which are steady; it has only been given the ten features. The confusion matrix shows that the family information is already encoded in those features in a way that is dramatically stronger than the per-workload information. That is exactly what one would expect if the phasic and steady classes are genuinely distinct signature modes rather than arbitrary labels.

SECTION 10

Honest limits

The result above is not a claim of universal workload classification. Five caveats should be acknowledged when this finding is consumed downstream.

  • The eleven workloads are not a population sample. They are the clean exemplars captured in v3. Any workload that mixes phasic and steady behaviour (for example, a database with periodic checkpoints) is not represented and may behave unpredictably under this classifier.
  • The sample size per workload is small. Each workload contributes 9 to 12 cells. The per-class metrics on the eleven-way task have wide confidence intervals, and the absolute numbers should not be over-interpreted at the second decimal place. The binary task is more robust to small samples because each cell contributes to one of only two classes, giving 54 phasic and 64 steady cells.
  • The Random Forest's perfect binary score is unlikely to hold on completely unseen workload types. What it demonstrates is that the phasic-versus-steady boundary is sharp in the feature space defined by the Plan 03 sweep, on the workloads in this dataset. A new workload from outside the captured set could in principle land near the boundary; this is a question for the next dataset.
  • The replicate-level holdout is strict, but it does not test cross-time-of-day or cross-host generalisation. All four replicates were captured on the same host, against the same guest, within a single day. Replicate-level variance reflects only the random initial state and the small inter-cell residue that D-82 cleaning leaves behind.
  • The features are derived; they are not the raw trajectories. A deep model trained directly on the raw active-page-fraction series might extract more discriminative within-family information. This is downstream work for any sequence-model upgrade, and is unrelated to the question this document answers.

SECTION 12

How to reproduce

The script that produces every number in this document is plan04_classify.py in the repository (the file name is historical; this is the classification phase, not the Plan 04 segmenter). The input is the combined sweep CSV produced by running the Plan 03 sweep over the merged v3_full and v3_ext bundles. The output is a JSON file with the same metrics, plus the ASCII summary that is printed to standard output.

$ python3 plan04_classify.py \ --sweep-csv _v3_combined/plan03/sweep.csv \ --window 8 --hop 4 \ --output _v3_combined/plan04_classification.json # Loads 118 cells across 11 workloads at W=8 H=4. # Replicate groups: [0, 1, 2, 3]. # Runs three classifiers under leave-one-replicate-out CV. # Reports binary and eleven-way metrics + confusion matrices. # Wall-clock: under 5 seconds on a laptop CPU.

The script depends only on scikit-learn version 1.7 or newer, NumPy, and the standard library. The JSON output schema is plan04.classification.v1 and contains, for each classifier, the accuracy, macro F1, confusion matrix, per-class precision/recall/F1, and the binary phasic-versus-steady metrics computed by remapping the predictions through the same family classifier that plan02_run._classify_workload uses.
