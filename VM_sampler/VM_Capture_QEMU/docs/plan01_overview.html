Plan 01 is the foundation. Before any per-family intervalMsec choice could be honest, the capture pipeline itself had to be calibrated. That meant nailing down the three time axes, fixing a 15× silent mis-calibration (bc missing), proving stationarity across the iv sweep, and removing the 0.5 s flush sleep for ~10 % wall-clock savings per snapshot. This page is the clean read-out of all of it.

SECTION 01

What this is

Plan 01 is the timing-instrumentation experiment. Before any of the later plans (interval tuning, window/hop, k-segmentation) could trust their inputs, the capture pipeline had to prove it could measure itself honestly. This page reads top-to-bottom and answers the question "what did Plan 01 actually do?".

Simple

Phase 1 of the thesis ran a memory-snapshot capture pipeline on a VM. Plan 01 added timestamps inside the producer at every step (suspend, pmemsave, resume) so we could measure how the pipeline actually behaves — and discovered a series of bugs that had been silently corrupting every previous experiment.

Technical

Plan 01 inserted six per-snapshot timestamps (t0..t5) into capture_producer_qemu_pmemsave.sh, plus a JSONL stream + backpressure events. Built three diagnostic tests (2a consumer isolation, 2b interval sweep, 2c flush sensitivity) plus a sanity orchestrator. Iteratively closed 7 theoretical mechanisms (i-vii) and 5 engineering bugs (D-H) across 5 rounds. Produced a stationary, calibrated, audit-trailed pipeline as input to Plan 02.

Plan 01 is the "instrument" experiment. Plan 02 is the "measurement" experiment. You cannot trust a measurement made by an uncalibrated instrument. That is why Plan 01 came first and consumed roughly 8 days of the project's 19-day timeline.

SECTION 04

The three time axes · Plan 01's core insight

Before Plan 01, "time" in this pipeline was conflated. There are three distinct axes; they look the same in code but mean different things in physics.

A
Guest-running interval
How long the VM was allowed to execute between snapshots. This is the knob — what intervalMsec sets.
measured: 0.125 s when iv=100ms · iv + 25ms producer overhead
B
Host wall-clock per snapshot
How long the capture cycle takes from suspend to resume. Dominated by pmemsave disk I/O.
measured: ~1.5 s post-fix · was ~9.25 s pre-bc
C
Analyzer Δt_frame
The spacing between samples as seen by the downstream analyzer. Must equal Axis A's measured value, not the configured one.
measured: 0.125 s · matches Axis A ✓
Why this matters

Before Plan 01, the analyzer assumed Δt_frame = 0.100 s (because intervalMsec was 100). Reality was 0.0066 s (bc bug). The analyzer's cepstral peaks were placed at frequencies 15× wrong on the spectrum, causing false confusion between workloads that should have separated easily. Fixing this one calibration alone may explain a lot of pre-Plan-01 classifier confusion.

The bug in one line

The producer shell script ran sleep "$(echo "scale=3; $intervalMsec/1000" | bc)". With bc not installed, the inner pipeline emitted empty stdout. sleep "" is a no-op (returns immediately). So intervalMsec set nothing.

SECTION 05

The four diagnostic tests

Plan 01 built four small instrumented scripts. Each isolates one specific question. None of them modifies the capture pipeline. They are diagnostic probes.

scriptquestion it answersmethod
sanity does the producer tick at intervalMsec? one 30 s producer run · idle Kali · measure guest_dt
2a how much variance is the consumer's fault? A/B · consumer-on vs consumer-killed · 60 s each
2b what's the pause-fraction at each iv? 4-cell sweep iv ∈ {100, 250, 500, 1000} ms
2c is the 0.5 s flush sleep necessary? A/B · flush-on vs flush-off · plus dump-integrity probe
Each test outputs one JSON. The JSON is the artifact of record. Per Plan-01 discipline, "no JSON, no claim". All round analysis cites specific JSON field names, never raw console output.

SECTION 09

The deliverable

Plan 01 produced three artifacts that the rest of the project leans on.

1 · A calibrated capture pipeline

Stationary host_dt at every iv level. Pause-fraction known and parameterized as a function of iv. Producer rolling-delete option (TIMING_SELF_CLEAN) eliminates accumulation drift. Sanity / 2a / 2b / 2c instrumentation scripts available for any future re-validation.

2 · The "REMOVE the 0.5 s flush" verdict

For production use (consumer keeps up with producer · normal phase-2 capture model), the 0.5 s flush sleep is overhead. Removing it saves ~10 % wall-clock per snapshot. Verified empirically in Round 5 with the orchestrator's automatic recommendation field reading "REMOVE the flush".

3 · The three-time-axes model

Documented in the HTML lab. Used by the downstream analysis to compute Δt_frame correctly (Axis C = measured Axis A, not configured iv). Will be cited in the thesis methodology chapter.

SECTION 10

Honest limits

Two things the team noted but did NOT close inside Plan 01. Both are tracked for downstream work.

Limit A · single-host bias

Simple

Every Plan 01 measurement happened on pcrserral. The pause-fraction curve, the pmemsave-flat 0.77 s, the bc-overhead 25 ms — all are properties of that one machine. A different host with a different disk / kernel / cpu would have different numbers.

Technical

Reproducibility metadata (kernel, qemu version, vm image sha256, git sha) is in schema v2 from Day-0 of Plan 02 specifically because of this. Future host migrations need to re-run sanity + 2b on the new host before claiming the per-family iv table still applies.

Limit B · idle-VM characterization only

Simple

Plan 01 measured the pipeline with an idle Kali VM. We never measured what happens when the VM is also running a memory-stressing workload at the same time. The pause-fraction sweep would shift; pmemsave dominance may change.

Technical

Plan 02 Step 2 ran workloads via SSH-launch for 90 cells and showed pause-fraction reproduces idle-VM measurements within 1 %. So in practice the idle-baseline holds under workload load. But this was empirical confirmation, not Plan 01's claim. Recorded for completeness.

SECTION 11

Common confusions

"Why is Plan 01 about instrumentation, not measurement?"

The naming is unfortunate. "Plan 01" comes from the tuning-plan series (01_instrumentation_logging_plan.md, 02_interval_tuning_experiment.md, ...). The 01 prefix means it is the first tuning plan in the series, and that first plan happens to be about getting the instrumentation right. Plan 02 is the first measurement plan that uses the instrumentation built by Plan 01.

"What does 'mechanism' vs 'bug' mean?"

Mechanism (lowercase Roman, i-vii) is a theoretical hypothesis for why the classifier might have confused workloads. They came from pre-Plan-01 confusion-matrix analysis and represent "things the analysis chain could be doing wrong." Bug (uppercase Latin, D-H) is a concrete engineering defect that was discovered during the rounds. Both are tracked separately so reading old reports doesn't confuse hypothesis with finding.

"What is the actual REMOVE-the-flush change?"

Inside the producer shell script, there was a sleep 0.5 after each pmemsave call (line 142 of capture_producer_qemu_pmemsave.sh). Plan 01 verified empirically that this sleep is unnecessary when the consumer keeps up with the producer (production model). Removing it saves ~0.5 s per snapshot. The flag TIMING_NO_FLUSH=1 already skips it; the team's recommendation is to make that the default.

"Why didn't we delete the flush already?"

Plan 01 produced the verdict (REMOVE). The actual line-deletion in the production producer is a separate operator action that didn't block any other plan. It will land alongside D-51 (analyzer-then-delete hook) when the producer is next touched.

"Did Plan 01 close ALL the bugs in the capture pipeline?"

All twelve known ones. There could be more that haven't been triggered yet. Schema v2's reproducibility metadata + the validator (plan02_validate_session.py) are the safety nets for catching future regressions early. The pattern is: any cell that fails C1-C5 surfaces the specific failure mode in notes.
