SECTION 01

The operator's three questions

operator · Q1

What exactly does Plan 03 not unblock? What is going to be Plan 05?

Plan 03 cannot raise the snap-throughput ceiling. Every cell in the v2 experiment got between 5 and 73 memory snapshots, with a median around 22. The theoretical target at iv=100 ms for a 300-second cell is 3000 snapshots. We got around 50 instead.

The reason is hardware physics. Every snap is a pmemsave command that writes the full 1 GiB of guest RAM to disk. Even on a fast SSD that takes between 300 milliseconds and 2 seconds. While that write is happening, the VM is paused. Then the B+3.1 helper has to read the old dump back, compute which pages changed, and delete it. So even when we ask for one snap every 100 milliseconds, real cadence is more like one every 3 to 15 seconds.

The consequence for Plan 03 is that the analyzer's sliding window cannot exceed the trajectory length. The original design doc wanted W × iv ≥ 4 × the workload's slowest rhythm (50 seconds for ransomware phases). At iv=500 ms that means W ≥ 400 snaps. No v2 cell has 400 snaps. So that rule is infeasible. Plan 03 relaxes it from 4× to 2× and documents the gap.

A future Plan 05 would fix the snap-throughput ceiling. Four directions, none of which are analyzer changes:

  • Faster storage (multi-NVMe RAID for higher write bandwidth)
  • Smaller VM RAM (256 MB instead of 1 GiB makes dumps 4× faster)
  • Incremental delta-encoded snapshots (eBPF page-fault tracing or QEMU's own dirty-page API)
  • A different capture mechanism entirely (sample-based access tracing, no full dumps)

Plan 03's job is "do the best with what we got." Plan 05 would change what we get.

operator · Q2

Why are we calling them "deltas"? What does that even mean?

Two unrelated meanings of "delta" collide in this codebase. They share a word but nothing else.

Meaning A — project-management jargon. When the team reviews an existing plan and proposes incremental changes, those changes are labelled "deltas." The word is shorthand for "what is different from the baseline." The B+3 to B+3.1 review proposed four such changes, labelled Δ-1 through Δ-4. The Plan 03 review proposed five, labelled Δ-1 through Δ-5. These are bullet points in a planning document. No mathematics involved.

Meaning B — mathematical distance. There is a Rust binary in the codebase called live_delta_calc. It takes pairs of memory dumps and computes cosine distance plus Hamming distance between corresponding 4 KiB pages. The output is, literally, deltas — numerical differences between pages. This is Phase 1's terminology, used in the original run_files_controlled.py pipeline.

Plan 03's five deltas are five edits to a planning document. They are not measurements, not metrics, and not anything the Rust binary touches. Counting them:

  1. Δ-1 — narrow window range from {32, 64, 128, 256, 512} to {8, 16, 32, 64}.
  2. Δ-2 — recalibrate the acceptance thresholds based on v2 measurements.
  3. Δ-3 — feed the cepstrum module a 1-D APF series reshaped as [T, 1] instead of forking the module.
  4. Δ-4 — add the C7 operational claim to the validator plus emit plan03_recommendation.json.
  5. Δ-5 — v2 regression guard so the chosen (window, hop) cannot silently degrade the v2 baseline.
operator · Q3

Are we calculating the same cosine / Hamming deltas as before? What is going on?

No. Two parallel pipelines exist, and we switched from one to the other for v2.

Pipeline A (Phase 1). Producer dumps RAW memory. A consumer process (the Rust live_delta_calc binary) reads consecutive dumps and computes per-page cosine distance and Hamming distance — both continuous values in [0, 1] — for every 4 KiB page in the dump. The output is a 2-D matrix [T snaps, N pages] where N is roughly 262144 pages for a 1 GiB guest. Downstream analyzers (PLV, MSC, cepstrum) operate on this 2-D matrix. Disk-expensive: keeping the full dumps means about 60 GiB per long cell, around 5 TiB for 90 cells. Infeasible on our server (around 145 GiB free).

Pipeline B (v2 / B+3.1). Producer dumps RAW memory same as before. A Python helper (plan02_apf_helper.py) reads consecutive dumps with numpy.memmap, computes a single number per pair: the Active Page Fraction (APF), defined as the fraction of 4 KiB pages whose contents differ at all between snap N and snap N+1. Then the helper deletes the previous dump. Persistent output is a 1-D float series, one APF value per pair. Disk usage stays bounded at about 2-3 GiB regardless of cell duration. For 90 cells the total persistent footprint is 4.7 MB.

Trade-off. APF gives a coarser memory-activity signature than cosine / Hamming. Cosine and Hamming say "for each page, how similar are the bytes?" APF says "did this page get touched at all, yes or no?" APF is enough for the thesis's separation claim (ransom CV is 28–140× workingset CV at every iv, F1 saturates from iv=250 upward). It is not enough for a per-page sub-signature claim, which would require Pipeline A.

Plan 03 operates purely on Pipeline B's 1-D APF trajectory. It sweeps analyzer (window, hop) settings over those trajectories. It does not touch cosine or Hamming. The Rust binary is dormant in the repository and would re-enter the pipeline if a future plan solves the disk-budget problem.

SECTION 02

What APF actually is — foundational concepts

advisor · Q4

In one sentence, what is APF?

APF (Active Page Fraction) is a single floating-point number per pair of consecutive memory snapshots, equal to the fraction of 4 KiB pages whose bytes changed at all between the two snapshots.

If a guest VM has 1 GiB of RAM, that's 262144 pages of 4 KiB each. If between snap-N and snap-N+1 exactly 65536 pages had any byte change, APF for that pair is 65536 ÷ 262144 = 0.25. The math is (pages_where_old_bytes != new_bytes) / total_pages. A page that flipped one bit and a page that was completely rewritten count the same.

advisor · Q5

Why a binary "changed or not" rather than a continuous distance? Aren't you throwing away information?

Yes, deliberately. Three reasons.

  • Disk budget. Continuous per-page distance (cosine / Hamming) requires keeping every full memory dump. That is 5 TiB for a 90-cell run. Binary "changed or not" needs only the previous dump available momentarily for the diff, then deletes it. Total persistent footprint: 4.7 MB instead of 5 TiB. Roughly a million-times reduction.
  • Sufficiency. The thesis's central claim is that phasic workloads (ransomware encryption phases) produce a recognizably different memory-activity signature than steady workloads (cyclic working-set churn). At the workload-class level that distinction is captured cleanly by APF alone. Ransom CV across the v2 cells is 28–140× workingset CV at every sampling interval. Continuous distances would refine the signal further, but they are not necessary to make the separation claim.
  • Compute speed. Binary diff with numpy.memmap on a 1 GiB dump completes in 300–500 milliseconds. Per-page cosine plus Hamming with the Rust binary is roughly 1.5-3 seconds. Faster diff means lower contention with the next pmemsave, which means more snaps per cell, which means a longer effective trajectory.

If the thesis ever needs to make a finer claim — for example "ransomware family A produces a different per-page distribution than family B" — we would have to re-introduce cosine / Hamming and pay the disk cost.

advisor · Q6

What exactly is a "snap" or "snapshot" here?

A snap is the output of one pmemsave call against the QEMU monitor socket. pmemsave dumps the entire guest physical memory to a file on host disk, byte for byte. For a 1 GiB guest the dump file is exactly 1 GiB.

During the dump the VM is paused (QEMU's stop / cont primitives are issued around pmemsave). Pause duration is roughly equal to dump duration, 300 milliseconds to 2 seconds. From the analyzer's point of view a snap is a single "frozen frame" of the guest's memory at one host-time instant.

The full lifecycle of one snap-pair under B+3.1 is:

t=0 virsh suspend (guest stops) t=0+ pmemsave -> /var/lib/libvirt/qemu/dump/snap_N.raw t=A pmemsave returns (~0.3-2 s wall-clock) t=A+ virsh resume (guest runs again) t=A+ async helper memmap-diffs snap_N-1 vs snap_N t=B APF written to apf_trajectory.jsonl, snap_N-1 deleted t=A producer waits until t=A + iv_ms, then loops to t=0 again
advisor · Q7

What does F1 mean in this context? It's a classification metric, but you're not classifying memory pages.

F1 here measures how well the APF trajectory's automatic "boundary detector" matches the ground-truth phase boundaries reported by the workload binary itself.

The ransomware workload (sandbox_ransom_batched) emits explicit phase-transition markers to its stderr during execution: "PHASE: encryption_start", "PHASE: encryption_end", and so on. Those markers are timestamped against the host clock. The host orchestrator captures them in workload_stderr.log per cell. We then know, with ground truth, at which snap-index in the APF trajectory each true boundary occurred.

Separately, the analyzer runs a simple boundary-detection heuristic on the APF series itself — "wherever |APF[t] - APF[t-1]| exceeds a median-plus-1.5σ threshold, declare a boundary." This gives a set of predicted boundary indices.

F1 is then the standard precision-recall F1 of predicted boundaries against true boundaries. F1=1.0 means every true boundary was detected and no false positives were declared. F1=0.0 means either no true boundaries were detected (all false negatives) or every prediction was wrong.

For the working-set workload there are no phase boundaries by design — it is steady-state. F1 is reported for it as well, but it is essentially noise (no true positives possible, so F1 is artifact of the detector's false-positive rate).

advisor · Q8

What does CV mean here?

CV is the standard coefficient of variation of the APF series: standard deviation divided by mean. It's a unitless measure of "how spiky" the trajectory is.

A steady workload that touches roughly the same memory at every snap produces an APF trajectory that hovers around its mean — CV is small, on the order of 0.02–0.10. A phasic workload like ransomware has long quiet periods (APF near 0) punctuated by encryption bursts (APF spikes to 0.4-0.5), so the std is large relative to the mean — CV is large, on the order of 2–5.

The thesis taxonomy claim is: ransom CV is 28–140× workingset CV at every iv (ratios computed from the v2 medians). That gap is the empirical separator between "phasic" and "steady" workload classes.

SECTION 03

Pipeline architecture

advisor · Q9

Why QEMU specifically? Why not VMware, KVM directly, or a container?

QEMU provides the cleanest path to a synchronous, full-memory snapshot from the host, via the QEMU monitor's pmemsave command. KVM (the kernel hypervisor) doesn't expose a memory-dump primitive to userspace by itself; it works in concert with QEMU.

VMware also has snapshot capabilities, but the API is proprietary, the licensing is restrictive for academic work, and the snapshot semantics are heavier (full VM state, not just memory).

Containers (Docker, podman, LXC) share the host's memory and don't have a clean "VM memory" abstraction to dump. You would have to sample /proc/<pid>/pagemap + /proc/<pid>/maps, which gives a fundamentally different signal (per-process, not whole-system). The thesis's whole-system memory model needs the VM-level abstraction.

The choice was driven by: licensing (QEMU is GPL, freely usable in academic publication), instrumentation API (pmemsave is documented and stable), and analytical model (whole-system memory is the unit of analysis).

advisor · Q10

Why does the VM have 1 GiB of RAM? Doesn't that affect the results?

1 GiB is the smallest RAM size where the chosen workloads can run cleanly. The ransomware workload reads, encrypts, and writes a set of files; with 256 MB it would either trigger guest-side swapping (which would dominate the memory signal with paging activity, not encryption) or fail to load the workload binary. The working-set sweep workload deliberately allocates 256 MB of working set, so a 256 MB guest would have no room for the OS plus the workload simultaneously.

Yes, RAM size affects results in a quantifiable way: every pmemsave is one full RAM dump, so disk write time scales linearly with RAM. A 256 MB guest would dump in roughly 80 milliseconds and give us ~4× more snaps per cell. Trade-off discussed in question Q1 / Plan 05 future work.

For the thesis claim of "phasic vs steady are separable at the memory-signature level," 1 GiB is sufficient and reproducible. For a productization story, a smaller guest with the same workload structure would give finer temporal resolution.

advisor · Q11

Why pmemsave instead of QEMU's dirty-page tracking API, which would tell you which pages changed without dumping the full memory?

Dirty-page tracking in QEMU is built for live migration, not for offline analysis. The API exists (QEMU_VM_SECTION_FULL plus the migration bitmap), but it has several limitations for our use case:

  • It samples, doesn't snapshot. The bitmap is updated continuously while the guest runs; reading it gives "which pages have been dirtied since the last reset" but you cannot freeze the page contents at a precise instant the way pmemsave does.
  • Tracking has its own overhead. Enabling dirty bitmaps requires the guest's EPT (extended page table) to mark pages as read-only on first access, causing a fault on the first write. That perturbs the workload itself, which contaminates the signal we are trying to measure.
  • The bitmap is just one bit per page. APF computed from pmemsave + XOR-diff is mathematically equivalent in information content (also one bit per page: "changed or not"), but with the option to upgrade to cosine / Hamming later without changing the producer side. Dirty bitmaps lose that upgrade path.

For a Plan 05 incremental-snapshot direction, dirty-page tracking is exactly the right primitive. For the current v1 / v2 / Plan 03 work, pmemsave is the right primitive because it gives us flexibility we may later need.

SECTION 04

Methodology defense

advisor · Q13

Why only two workloads? An attacker has hundreds of behaviors. Don't you need a broader sample?

Two workloads is the minimum to prove a separation between two classes. The thesis claim is not "we can detect any ransomware" or "we can detect every working-set behavior." The claim is "phasic memory signatures and steady memory signatures are empirically separable at the active-page-fraction level."

To prove that claim you need (a) at least one representative of each class and (b) measurements that distinguish them. We have both. The 90-cell sweep across 5 sampling intervals and 3 durations shows the separation is robust across capture parameters, not just one lucky setting.

What two-workload scope does NOT prove:

  • That every ransomware family produces this signature. Only sandbox_ransom_batched was tested.
  • That every steady workload looks like working-set sweep. Different memory-access patterns (sequential scan, pointer chase, etc.) would need to be measured separately.
  • That a real-world adversary cannot mimic the steady signature to evade detection.

Broadening to more workloads is appropriate follow-up work, but it is a separate research question from "can we separate the two canonical classes." That separation question is what the thesis answers.

advisor · Q14

How do you defend the iv recommendation table against the criticism that you cherry-picked it?

Three lines of defense.

  • Statistical pre-registration. The acceptance gates and decision rule were specified in the Plan 03 proposal before the sweep ran. The team is on record (in d51_bplus3_improved_proposal.html and the Plan 03 proposal) for the gates G1-G5. Picking a winner that does not pass the gates is publicly visible as a deviation.
  • Multiple-comparisons correction. Holm-Bonferroni FWER applied per workload (m=12 tests). Multi-test correction reduces the false-positive rate from the naive 0.05 per test to 0.05 family-wise. So winning a single (window, hop) by chance is much harder than it looks.
  • v2 regression guard. The chosen (window, hop) must not push the workload's median metric below the v2 baseline by more than 0.05 absolute. That is explicitly checked in the decision rule. A combo can fail the gates by degrading; it cannot win by happening to look slightly better on a quirky subset.

The Plan 03 proposal also includes a Simpson's-paradox guard: if the pooled winner across iv strata differs from any per-iv stratum winner, the aggregator refuses to auto-pick and demands explicit operator sign-off. That blocks the most common cherry-picking failure mode.

advisor · Q15

What if the phase markers from the ransomware binary are wrong or mis-aligned with the snapshot timestamps?

This is a real risk and the team flagged it explicitly during the v2 review. Four ransom cells (d84c750f0590, 01b7bc77c211, plus two more at iv=500 / 1000) have F1=0 despite plentiful snaps (41 and 57 snaps respectively). The data scientist flagged these as likely phase-marker misalignment rather than analyzer failure.

The mechanism is: the ransomware binary emits phase markers to stderr on the guest side. They are captured by the host orchestrator via SSH. The host clock and the guest clock are NTP-synchronized but not phase-locked. A skew of even a few hundred milliseconds is enough to mis-place a phase marker by one or two snap indices, which destroys F1 for short cells.

Plan 03's response is to flag these cells in the recommendation output (simpson_paradox_warning / outlier_flag) and to weight median statistics over mean statistics, since the median is robust to a handful of misaligned cells.

A real audit of the marker pipeline (host-clock to guest-clock skew measurement, marker arrival time vs guest emission time) is filed as Plan-03-adjacent follow-up work. It does not block the iv recommendation, because the recommendation rests on the median and the median is unchanged by 2 of 9 mislabelled cells.

advisor · Q16

Why these specific iv values (100, 250, 500, 1000, 2000 ms)? Why not a finer grid?

The values are logarithmically spaced because the expected effect of iv on the signal is multiplicative, not additive. At iv=100 ms versus iv=200 ms you see a roughly 2× difference in temporal resolution. At iv=1000 ms versus iv=1100 ms you see a 10% difference. Log spacing gives you uniform coverage in the regime where the effect actually lives.

The lower bound (100 ms) is set by hardware: pmemsave writes 1 GiB. On our disk that takes at least ~300 ms in the best case, so iv below ~300 ms cannot physically be sustained. iv=100 is in the sweep specifically to confirm that this hardware floor produces a measurable signal collapse (which it does — ransom F1 drops to 0 at iv=100).

The upper bound (2000 ms) is set by Nyquist: detecting a 50-second ransom phase requires sampling at ≤ 25 seconds per sample. We push past that comfortably (2000 ms = 25× margin) to confirm the signal survives.

A finer grid (every 100 ms from 100 to 2000) would multiply the experiment cost by 4× (20 iv values vs 5) without changing the conclusion: the data already shows that iv=250 through iv=2000 are statistically tied for ransom F1 (pairwise p > 0.96 between every pair), and the only meaningful break is at iv=100. A finer grid would replicate that finding at higher cost. Not informative.

SECTION 05

Statistical defense

advisor · Q17

n=3 replicates per cell. n=9 per (workload, iv). Isn't that statistically thin?

Yes, and we say so explicitly in the proposal docs. Three observations.

  • For the ranking claim ("iv=500 is better than iv=100 for ransom F1"), n=9 within (workload, iv) is sufficient because the effect size is large. Kruskal-Wallis on F1 across iv gives H=14.61 and p=0.0056. The effect is driven entirely by the iv=100 collapse; every pairwise test between iv=100 and any other iv has p < 0.05 even after Holm-Bonferroni correction. The signal is strong relative to the noise at this n.
  • For the absolute claim ("ransom F1 at iv=500 is 0.80 ± 0.35"), n=9 gives a 95% confidence interval of roughly ±0.27. That is wide. We do not report this as a precise number; we report it as a distribution with median and IQR. Readers who interpret the median as a precise point estimate are over-interpreting.
  • For the workingset claim ("workingset CV is stable across iv"), K-W gives p=0.648 — not significant. We say so explicitly: at n=9 we lack statistical power to distinguish CV across iv strata. The recommendation of iv=250 for workingset is based on tail behavior (cleanest IQR) and practical considerations (lowest disk pressure), not on a significant difference in central tendency.

A power calculation against the observed within-group std (~0.17 for CV, ~0.39 for F1) suggests roughly 20 replicates would be needed to detect a 0.05 absolute difference at 80% power. n=3 was a feasibility trade-off; n=20 would have made the v2 sweep take a week instead of overnight.

advisor · Q18

Aren't the three replicates within a cell going to be correlated? Same VM, same time of day, same kernel state.

Yes, and the data scientist explicitly flagged this in the v2 review. The 3 replicates of (workload, iv, d) run sequentially on the same VM with the same configuration. They share kernel state, page-cache state, and possibly time-of-day patterns. Treating them as 3 independent draws in a Kruskal-Wallis test inflates the test statistic.

Mitigation adopted in Plan 03's statistical design: collapse the 3 replicates to a single median value per (workload, iv, d) cell. That gives n=15 per workload at the duration level (5 ivs × 3 durations) instead of the naive n=45. Conservative but honest. The headline claim is unchanged (iv=100 collapse is still significant after this correction).

advisor · Q19

F1 is bimodal in your ransom data (mass at 0.0 and at 1.0). Doesn't that violate the assumptions of most statistical tests?

It violates the assumptions of parametric tests (t-test, ANOVA) which assume approximate normality. That is why the statistical design uses non-parametric tests throughout: Kruskal-Wallis, Mann-Whitney U, Friedman, Wilcoxon signed-rank. These rank-based tests do not assume any particular distribution shape.

Specifically for the F1 bimodality, we also report two summary statistics jointly: the median (which is robust to bimodality) and the proportion P(F1 ≥ 0.8). For the v2 ransom iv=500 cells the median is 1.0 and P(F1 ≥ 0.8) is 0.667. Reporting both lets the reader see "the median is at the upper mode but a third of cells fall in the lower mode" rather than hiding the bimodality behind a single number.

For confidence intervals we use BCa (bias-corrected and accelerated) bootstrap, which makes no distributional assumption and handles the boundary mass at 0 and 1 correctly. Standard t-based CIs would be misleading here because they assume symmetric variance which the data does not have.

advisor · Q20

How do you guard against p-hacking?

Three structural safeguards, plus one cultural one.

  • Pre-registered hypotheses. The acceptance gates (G1-G5 for Plan 03) and the iv recommendation criteria (for v2) were specified in published proposal documents (d51_bplus3_improved_proposal.html, plan03_window_hop_proposal.html) before any analysis ran. They are dated by commit timestamps in git history.
  • Multiple-comparisons correction. Holm-Bonferroni per workload, applied to all pairwise tests. Limits the family-wise false-positive rate at 0.05.
  • Regression guards. The chosen winner for Plan 03 must not degrade the v2 baseline. So even if a (window, hop) happened to look good on some sub-stratum, it cannot win if it sacrifices a metric the previous plan established.
  • Cultural. The week-long simulated team review (PM, SA, EN, ML, XD, DE, DS, EE) is on record in the HTML proposals. Disagreements and their resolutions are documented. A reader can read the disputes section of each proposal to see what was contested and how it was resolved. Cherry-picking would have left no audit trail.
SECTION 06

Threat model + workloads

advisor · Q21

What's the threat model? Who is the adversary and what are they trying to do?

The threat model is host-based detection of ransomware encryption activity inside a Type 1 hypervisor environment (host = QEMU/KVM on Linux, guest = a Windows or Linux VM that may or may not be compromised). The adversary is malware running inside the guest with no awareness that the host is observing its memory.

What the host can see: full guest physical memory at any moment, via QEMU's monitor interface. The host has root-equivalent access to the hypervisor.

What the host cannot see: contents of CPU registers between snapshots, contents of cache, decisions the malware made between snapshots (only the memory side effects are visible).

What the adversary cannot easily do (in this model): detect that the host is sampling memory. pmemsave works through hardware-mediated VM exits but does not signal to the guest that a dump occurred. The brief VM pause (300 ms - 2 s) is theoretically detectable via a timing oracle inside the guest, but exploiting that to evade detection requires the malware to have a counter accurate enough to notice and to alter its behavior accordingly — a non-trivial adversary upgrade.

The detection target: distinguish "this VM is currently running ransomware-like memory activity" from "this VM is running normal application memory activity." The thesis demonstrates this distinction is achievable at the APF level for two canonical workload classes.

advisor · Q22

Could a real adversary evade this by changing its memory access pattern?

Yes, in principle. The thesis claim is "we can separate the two canonical workload classes." It is not "no adversary can evade us." Two specific evasion routes are plausible:

  • Slow encryption (anti-phasic). If the adversary encrypts at a rate slow enough that each pmemsave sample sees only a small fraction of the working set changed, the APF spike characteristic of phasic encryption disappears. CV drops toward the steady-workload range. Cost to the adversary: encryption now takes hours instead of minutes — the attack is slower, possibly to the point of being interrupted by other defenses (network detection, behavioral analytics) before completion. There is a trade-off, not a free evasion.
  • Encryption interleaved with decoy steady activity. The adversary periodically touches a large working set to mask the encryption phases. Result: the APF series has both spiky and steady components; CV is intermediate. The naive phasic-vs-steady classifier would mis-classify. Cost to the adversary: extra memory bandwidth, easier to detect via other side channels (page-cache thrashing, CPU pinning).

Defending against these adaptive evasions is future work. The thesis's contribution is the measurement and analysis framework that makes "what does ransomware memory activity look like" a quantifiable question rather than an intuition.

advisor · Q23

Why those two specific workload binaries? What's sandbox_ransom_batched actually doing?

sandbox_ransom_batched is a research-grade ransomware simulation. It:

  • Walks a fixed directory tree on the guest's filesystem.
  • For each file in scope, reads it into memory, encrypts the bytes with AES-256-CBC, writes the encrypted bytes back to disk.
  • Runs in "rounds" where each round processes a batch of files (default 1 round = phase = ~10 seconds of activity).
  • Emits explicit PHASE markers to stderr before and after each round for ground-truth alignment.
  • Does NOT perform any actual network exfiltration or cross-host activity (academic sandbox).

mem_workingset_sweep_v2 is a synthetic steady-state probe. It:

  • Allocates a fixed working set (default 256 MB) of contiguous memory.
  • Walks that memory in a cyclic pattern, reading and writing pages at a rate that keeps the working set hot.
  • Maintains constant memory access rate for the full cell duration.
  • Does NOT emit phase markers — it has no phases by design.

The pair is chosen to be the cleanest possible representatives of the phasic and steady classes. Real-world malware will be messier than sandbox_ransom_batched, and real-world applications will be more variable than mem_workingset_sweep_v2. The thesis demonstrates the technique on clean exemplars; downstream work would apply it to messier real samples.

advisor · Q24

Why measure memory and not CPU? Network? Disk?

Four reasons memory was chosen as the primary signal.

  • Universality. Every computation touches memory. CPU monitoring (via performance counters) is rich but architecture-specific. Disk monitoring catches only workloads that write to disk. Network monitoring catches only network-active workloads. Memory is the common substrate.
  • Anti-evasion property. A guest workload can avoid touching disk (run from a RAM-disk) or avoid the network (operate locally), but it cannot run without using memory. Memory monitoring catches workloads that would be invisible to other channels.
  • Granularity. The page-level (4 KiB) granularity gives a useful intermediate level between the per-instruction CPU signal (too noisy) and the per-system-call disk / network signal (too coarse).
  • VM-friendly. The hypervisor already mediates memory access between guest and host hardware. Adding observation is a hypervisor change, not a guest change — the guest does not need cooperation or instrumentation.

Multi-modal monitoring (memory + CPU + disk + network) is a natural extension. The thesis scope is "what does memory alone reveal?" which is a meaningful question because it sets the floor for what multi-modal can achieve.

SECTION 07

Scaling + production

advisor · Q25

Could this scale to 1000 VMs or to a data-center-scale deployment?

Not as currently engineered. Three bottlenecks would surface.

  • Disk bandwidth. One VM at 1 GiB RAM with iv=1000 ms generates 1 GB/s of dump traffic during active capture. 1000 VMs would need 1 TB/s of write bandwidth. Even a high-end NVMe RAID array is 30-50 GB/s. Need either far fewer VMs sampled at any one time (sample-and-rotate), much smaller VM RAM, or incremental-snapshot Plan 05 work.
  • Trailing-helper race at scale. Plan 02's Bug O analysis estimated 22 out of 1000 cells would tail-miss on the original 30 s barrier. The Bug O fix (progressive idle timeout) reduces this; at 1000 cells the residual rate is probably 1-2 cells. Acceptable but not zero.
  • Validator throughput. The C1-C7 claim evaluator iterates per cell. Validation of 1000 cells takes minutes, not hours, but the validator was not designed for streaming live results — it expects all cells to be complete before validation runs. A live-detection deployment would need a streaming validator.

For a "monitor 10 VMs at once on a single host" deployment, the current pipeline is feasible after Plan 05 fixes the snap-throughput ceiling. For "1000 VMs across a data center" you would need a different architecture: distributed capture nodes, central aggregation, and the analyzer running in streaming mode rather than batch.

advisor · Q26

What is the latency from "attack starts" to "detection fires"?

End-to-end latency under the current pipeline at iv=500 ms (ransom recommendation):

stagetypical time
Attack onset (in-guest)t = 0
First snap pair containing attack activityt = iv = 500 ms (worst case 1000 ms if attack starts just after a snap)
APF computed for that pair+ 300 ms (helper memmap diff)
Window of W=16 acquires 8 new snaps (for boundary detection)+ 8 × iv = 4 s
F1 boundary detector fires on the spike+ 50 ms
Total cold-start detection latency~4.85 seconds

For a ransomware attack that encrypts hundreds of files over tens of seconds, sub-five-second detection latency is operationally useful. For an attack that completes in 1-2 seconds (a single small file encrypted with a script-kiddie's tool) the current pipeline would miss it because the entire phase is shorter than one analyzer window.

Sub-second detection would require either smaller windows (W < 8, which Plan 03 explicitly excludes due to cepstral noise floor) or higher iv (which Plan 05 addresses).

advisor · Q27

What is the false-positive rate? If you deploy this in production, how often will it alarm on benign activity?

The thesis does not directly measure false-positive rate, because the working-set workload was specifically chosen to be a "non-phasic" reference, not a representative sample of all benign workloads. The fair statement is: "ransom and workingset are separable; we have not characterized the false-positive rate against arbitrary benign software."

What the v2 data does say: workingset CV across all 45 cells stays below 0.15 (the loose acceptance threshold) in 100% of cases and below 0.05 in 85% of cases. Ransom CV stays above 0.5 in 95% of cases (excluding iv=100). So a CV-based classifier with a threshold somewhere in the 0.1-0.5 range would give roughly zero false positives on workingset and roughly 5% false negatives on ransom.

This is a per-cell rate, not a per-time rate. In a deployment that evaluates the metric every minute, a 5% false-negative cell rate corresponds to an attack taking on average about 12 minutes to be reliably detected. Adjustable by tuning the threshold or by requiring K consecutive positive cells.

Characterizing the false-positive rate against a diverse set of real applications (Postgres under load, Chrome, a video encoder, a build server) is a critical productization step and was not done as part of the thesis scope.

Future work + Plan-N

advisor · Q28

What would Plan 04, 05, 06 actually be?

Each plan addresses a distinct layer of the pipeline:

  • Plan 04 — segmenter tuning. Given an APF trajectory and a Plan 03 (window, hop), how does the segmenter decide where one "phase" ends and another begins? Today the segmenter is a stub (median + 1.5σ threshold). A trained segmenter (HMM, change-point detector, or a small neural model) would improve F1 numbers absolutely while leaving the iv ranking stable.
  • Plan 05 — snap-throughput ceiling. Discussed above (Q1). The hardware-limited 5-73 snaps per cell becomes the bottleneck for window-size selection. Plan 05 raises that ceiling through faster storage, smaller guest RAM, or incremental snapshots.
  • Plan 06 — cosine / Hamming re-introduction. If Plan 05 solves the disk budget, the original Phase 1 cosine and Hamming per-page distance pipeline can come back. The richer 2-D [T, N] signal would enable per-page sub-classification (which memory regions are oscillating, which malware family the rhythm matches).
  • Plan 07 — multi-workload broadening. Add workloads beyond the two canonical exemplars: Postgres, Redis, Chrome, kernel build, scientific compute, web server. Characterize where each falls in the phasic-steady spectrum. Build a population baseline for false-positive analysis.
  • Plan 08 — adversarial robustness. Run the slow-encryption and decoy-interleaved evasion strategies. Measure detection degradation. Propose multi-modal counters.

These are notional. Only Plan 01, 02, and 03 are formally proposed in the codebase today.

advisor · Q29

What machine learning would you put on top of this?

Three layers, in increasing sophistication.

  • Threshold classifier. A single threshold on CV (or on F1, or on a small linear combination). Already implicit in the thesis's recommendation table. No training data required. Detects "phasic activity present yes/no."
  • Sequence model on the APF trajectory. An LSTM, GRU, or small Transformer trained on (APF series, phase label) pairs across many workloads. Predicts not just "phasic / steady" but "what phase is the workload currently in." Plan 03's (window, hop) recommendation gives the input feature window size.
  • Multi-modal model. APF as one feature stream alongside disk I/O rate, network bytes, CPU PMU counters. Cross-modal correlations are stronger than any single channel. Requires extending the capture pipeline to other observation modalities — not a thesis-scope deliverable.

The thesis sticks at the "threshold classifier" layer because the central claim is "the signature itself is separable." Layers 2 and 3 are downstream engineering.

advisor · Q30

Could this run on real hosts, not just VMs? The threat model needs to address real attacks.

Yes, with a different memory-access primitive. The QEMU-specific pmemsave mechanism does not apply to bare-metal hosts. Three plausible substitutes:

  • Linux /proc/<pid>/pagemap + /proc/<pid>/maps + /proc/<pid>/mem. Per-process memory observation from a privileged userspace agent. Coarser than the whole-system view, but real-system-applicable.
  • eBPF page-fault tracing. Hooks into the kernel's page-fault handler to log every fault. Gives a fine-grained access pattern per page. Performance overhead is the concern: page faults are frequent, logging every one is expensive.
  • Intel Processor Trace + memory taint. Hardware-supported instruction tracing combined with memory-access logging. Requires recent Intel CPUs and significant analyzer infrastructure. Highest fidelity, highest cost.

The thesis's VM-focused work is a controlled environment in which the signal can be cleanly measured. Translating to bare metal is downstream engineering of a different research question: "does the same memory signal survive in a less-controlled environment." Not blocked by anything in the thesis; just out of scope.
