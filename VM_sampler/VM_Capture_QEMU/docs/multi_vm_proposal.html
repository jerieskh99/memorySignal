Team reviewed three multi-VM topologies against four criteria: speed-up, honesty, production-match, and design risk. Recommendation: stay single-VM for the Plan-02 v2 sub-pilot. The ~30 % speed-up does not justify the cross-VM confound for an experiment whose v1 deliverable is already published. Conditional re-eval if a future question changes (multi-tenant attack scenarios, large-N sweeps, host-contention robustness).

SECTION 01

Your question · as the team read it

You asked: should we run multiple VMs in parallel (or interleaved) on the same host to finish v2 faster, given that we only sample the VMs and they don't share guest RAM?

In one sentence

You want to know if a multi-VM topology can speed up data generation without making the per-family iv recommendation less trustworthy.

Formal restatement

Given an experiment whose per-cell measurement is dominated by host-side disk + scheduling time, and given that QEMU domains are memory-isolated, can we exploit host CPU/IO parallelism to reduce wall-clock for a 50-90 cell sub-pilot without introducing cross-cell confounds that bias the per-family iv selection?

Three topologies considered: (A) per-family allocation · 1 VM per family running its family's cells solo; (B) sampling-interleave · multiple VMs, sampled round-robin while others run; (C) single VM (current).

SECTION 04

Honesty risk

Cell independence is the core statistical assumption. Multi-VM violates it in subtle but real ways.

In plain English

If two cells run simultaneously, and the host is contended (disk, CPU, cache), then both cells' measurements include a shared "I was contended" signal. The pause-fraction we report for cell A no longer measures "cost of intervalMsec under workload A" — it measures "cost of intervalMsec under workload A given that workload B happened to be running on the same host at the same time".

That extra dependency means the recommendation table is now conditional: "use iv=500 ms for ransom, but only if there's no concurrent workingset workload". That is a different and weaker claim than what v1 makes.

Statistical framing

Cell measurements y_i have an additive cross-cell error ε_ij when cells i, j share host time. The cross-error covariance is non-zero whenever i and j overlap. Linear models fitted across cells will under-estimate variance (correlated residuals) and over-estimate effect sizes.

Mitigations exist (mixed-effects model with host_load_state as a random effect, or block-randomization where each block is "host-quiet" vs "host-loaded") but they multiply analysis complexity without changing the recommendation in most cases.

Speed-up math · real numbers, not hand-waves

v2 sub-pilot is the only relevant target. v1 is already in hand. Numbers from Step 2 timing.

Option A · per-family parallel

Assume 3 VMs in parallel (one per family bucket). Each handles ~1/3 of cells. Best-case wall-clock = Tseq / 3 = 2 h if there is no contention. Realistic with contention slowdown:

Best case (no contention)
2 h
Realistic (1.4× per-cell)
2.8 h
Bad host (2× per-cell)
4 h

So realistic speed-up is ~30-50 % (saving 2-3 h on a 6 h baseline). Not 3×.

Option B · sampling interleave

Doesn't save wall-clock. The orchestrator already runs cells sequentially. Interleaving samples across VMs does not change the total number of pmemsave calls; it just rearranges them. The total host disk-write workload is unchanged.

Option B's appeal is "per-VM pause-fraction drops because each VM runs free while the others are sampled". True per-VM, but the host wall-clock needed to complete the experiment is identical. Pause-fraction is not the bottleneck; host disk + scheduling is.
SECTION 06

Does multi-VM match production deployment?

In plain English

The Plan-02 iv table is for picking sample intervals when capturing real workloads on real hosts. In your thesis, how is each capture session going to run? If it's "one workload per host at a time", measure that. If it's "multiple workloads on the same host", measure that.

Right now your production model is the single-VM case. Phase-2 captures are designed to characterize one workload's signal, not a workload-mixture signal.

Deployment-fidelity argument

External validity requires the experimental conditions to match the target deployment. If the deployment is one VM at a time, the experiment must measure under that condition. Otherwise the iv table is correctly calibrated for a deployment that does not exist.

Counter-case: if the thesis includes a multi-tenant deployment claim (e.g. "this works in cloud environments where many workloads share a host"), then multi-VM measurement is required for validity. The thesis presently does not make that claim.

Production model · single-VM. The capture pipeline you've been calibrating is a single-VM tool. Plan-02 v1 numbers describe that tool. Multi-VM would describe a different tool you're not currently building.
SECTION 07

Per-agent verdicts

SA
Orchestrator currently assumes one VM. Multi-VM means N parallel orchestrators OR a redesigned single orchestrator with VM-multiplexing. The former is ~0 code (just launch in N tmux panes with N manifest dirs) but introduces N copies of the queue, dump dir, ssh target. The latter is a significant refactor. Both come with new failure modes (e.g. one VM dies, others continue silently). The smallest reliable design is N parallel single-orchestrator instances with strict CPU pinning per VM.
conditional
EN
pmemsave on QEMU is host-side disk-bound. Two pmemsave calls writing 1 GiB simultaneously will saturate most SSDs and absolutely saturate any HDD/NFS. host_dt becomes non-stationary, which is exactly the property Plan-02 spent 11 days establishing as stable. If you must, schedule pmemsave windows so they never overlap — but then the speed-up vanishes.
no
ML
The recommendation table is design-justified, not metric-driven. Multi-VM does not produce new metric data; it produces faster-but-noisier metric data. Faster + noisier = wash for a recommendation that was already going to be cross-checked with Plan-03 anyway. Multi-VM gives nothing the analysis chain needs.
no
XD
Cross-workload parallelism (option A across families) is a confound. Within-family replicate parallelism (option A within a single family) is recoverable — same workload, different replicates, controlled host load. But within-family parallelism only buys you 3× speed-up if you have 3 VMs, and your replicate count is 3, so the per-family wall-clock goes from 30 min to 10 min — saving 20 min per family for 6 families = 2 h saved, at the cost of running 3 VMs simultaneously. The math says don't bother.
no
DS
Cell independence is the easiest statistical assumption to violate and the hardest to recover from. Once you have correlated residuals, every linear test becomes suspect, and you end up needing mixed-effects models with two random effects (workload AND host_load_state). That is a lot of statistical machinery for a 30 % speed-up. Keep cells independent.
no
EE
Acceptance criteria for v2 require per-cell C1-C5 to pass. Multi-VM introduces a new claim (C6: "no concurrent VM contamination") that we have no instrument for. We could add such an instrument (e.g. correlate cell timing with concurrent host load) but that is exactly the kind of code we just rejected. The instrumentation budget is spent.
no
PM
v1 is shipped. v2 sub-pilot is ~6 h on a single VM. Multi-VM saves ~3 h at best, costs design + validity. Honesty risk is real and unmeasured in our setup. Production model is single-VM. Stay single-VM for v2. Re-evaluate if a future question changes the production model.
no

SECTION 09

When multi-VM WOULD make sense

Three scenarios where the verdict flips.

Scenario 1 · multi-tenant production claim

If the thesis ever claims "this signal works in cloud / multi-tenant hosts", multi-VM measurement is required for external validity. Single-VM data wouldn't support that claim. Different thesis scope.

Scenario 2 · 1000+ cell sweeps

If a future experiment requires hundreds of hours of capture (e.g. exhaustive workload-binary cross-product), parallelism becomes more attractive. At that scale the design cost of building multi-VM cleanly amortizes. v2 at ~6 h does not approach this threshold.

Scenario 3 · within-family replicate parallelism with strict isolation

If you ever need to run dozens of replicates of the SAME workload, running 3-4 VMs of that workload in parallel with strict CPU pinning + non-overlapping pmemsave windows is defensible. Same workload means no cross-workload confound. But the pmemsave-window scheduling code does not exist; building it is comparable to D-51 in effort.

None of these scenarios applies to v2. Park multi-VM as a future option; revisit when the question changes.
