SECTION 01

What this is

A single canonical overview of Plan 02. Built because the long-running HTML lab has 11 days of in-place edits and the term "Plan 02" appears in multiple states. This page replaces the question "what's going on?" with a clear answer.

Simple

Phase 1 of the thesis used one fixed sample interval for every workload. Different workloads have different natural rhythms. Plan 02 figured out the right sample interval per workload family by running 90 controlled captures across 5 interval values × 3 durations × 2 workloads × 3 replicates.

Technical

Plan 02 is a factorial controlled experiment over intervalMsec ∈ {100, 250, 500, 1000, 2000} ms. Outputs: per-family recommended iv that balances pause-fraction cost against workload-rhythm resolvability. v1 deliverable: iv_recommendations_v1.json.

Three things share the name "Plan 02". Knowing which one any sentence refers to is the only trick to reading the old docs.

SECTION 08

Honest limits

Three things v1 does NOT yet measure. None invalidates the table; each has a specific small fix.

Limit A · no F1 / CV measurement

Simple

Step 2 ran the workloads but did not keep the dump files (would have filled the disk). Without dumps we cannot compute the analyzer-side metrics. v1 picks are made from cost data + workload-design priors, not from a measured signal-quality vs iv interaction.

Technical

analyzer_outputs.{f1_phase, cv_workingset} are null across all 90 cells. Fix is D-51 (analyzer-then-delete hook) · ~150 LOC. Then run a smaller targeted sub-pilot to populate the metrics.

Limit B · mlock failure in workingset

Simple

The workingset_sweep_v2 workload binary asked the Kali VM kernel to pin 512 MiB of its memory; the kernel refused. The workload continued anyway but with some pages swappable. Its rhythm is mildly degraded as a result.

Technical

mlock(536870912) failed: Cannot allocate memory in every workingset stderr. Fix on VM side: raise RLIMIT_MEMLOCK, OR bump VM RAM, OR reduce --working-set-mb from 512 to 256. Documented in audit Day 11 D-50.

Limit C · canonical window too coarse

Simple

Even the longest cells (30 min) don't produce enough sliding windows at the Phase-1 default window=128 hop=64. That's a separate question: how big should the analyzer's window be? That is what Plan 03 will answer.

Technical

Max n_windows observed in Step 2: 16 at iv=100 ms, d=1800 s (1088 snaps). To clear 50 windows at (128, 64) needs ≥ 3328 snaps per cell — at iv=100 ms that is 7 min guest-running ≈ 86 min host per cell. Infeasible at 90-cell scale. Resolution: Plan 03 chooses smaller window/hop. C3 in the validator is informational because of this.

SECTION 10

Common confusions

"I thought we finished Plan 02 · why is it 'next' in the lab?"

The lab text says "next is D-51". D-51 is the upgrade from v1 to v2. v1 is already finished. "Plan 02" as a single concept has multiple deliverables; v1 is shipped, v2 is optional refinement.

"Do I have the per-family iv numbers?"

Yes. See Section 7 above. Six families mapped to iv ∈ {250, 500, 1000} ms. METHODOLOGY inherits from child workload.

"Why are recommendations 'design-justified' instead of 'measured'?"

Because Step 2 did not preserve dump files (would have filled disk). Without dumps, we cannot compute F1 / CV. The v1 picks use pause-fraction (measured) + each workload's intended rhythm (designed). D-51 closes that gap.

"Should I run D-51 before writing the thesis?"

No. D-51 produces stronger evidence but is unlikely to flip any pick. Write the chapter using v1; rerun D-51 in the background if you want belt-and-suspenders empirical backing.

"What about Plan 03?"

Plan 03 (window/hop tuning) is the next experiment, independent of v1 vs v2. C3 in the validator is direct evidence that the canonical (128, 64) is too coarse for our durations. Plan 03 is unblocked now; the prerequisite (calibrated capture pipeline) is in place.

What the numbers tell us

Simple

For the steady workload workingset_sweep at iv=100 ms, CV came back at 0.033 — that's the workload's memory rhythm wobbling by ~3 % around its average. Steady workloads should have low CV. This is exactly the expected signature.

For the phasic workload ransom_batched at iv=100 ms, F1 came back at 0.67 — the stub change-point detector found 4 of 6 phase boundaries. CV is 3.45 (very high) because APF spikes dramatically each phase transition. This is also the expected signature for a phasic workload.

Both signatures match what v1's design priors predicted. The 6 cells confirm the two-regime model.

Technical

workingset CV = 0.033 at iv=100 ms vs designed CV target ≤ 0.27 (D-5 noise floor calibration) — well below the FAIL line. Steady-state hypothesis confirmed.

ransom_batched F1 = 0.667 at iv=100 ms vs designed F1 target ≥ 0.70 — just under the strict line, comfortably above the 0.52 noise floor (also D-5). Phase detector finds boundaries; iv=100 ms is at the lower edge of usable resolution for this workload.

APF trajectory complete on every cell post-D-76 (sentinels parsed cleanly, 6/6 C6 PASS). All B+3.1 deltas verified empirically.

Why these 6 cells can be sufficient (without running full v2)

In plain English
  • The expensive thing was learning the iv-cost curve. v1 already did that (pause-fraction sweep across 5 iv levels × 90 cells). 6 more cells don't change that curve.
  • v1 picks were design-justified. The 6 cells tested 3 of the 5 iv levels against both regimes (steady, phasic). Both regimes showed the expected signatures. No surprises.
  • Full v2 would multiply replicates. 90 cells = 5 iv × 3 dur × 2 workloads × 3 reps. That gives statistical power for tight confidence intervals around F1/CV. With 6 cells we have point estimates only — but the point estimates land where v1 predicted.
  • For a thesis chapter, point estimates + design rationale + 6-cell confirmation is publishable. Full 90-cell v2 strengthens the claim but does not flip any iv pick.
Technical
  • v1 recommendation table is mode-grounded (steady → slow iv, phasic → mid iv). 6-cell sub-pilot empirically validates the steady-vs-phasic discrimination at 3 iv levels.
  • Statistical power: with n=1 per (workload, iv) cell, no CI estimable. With n=3 (per a full v2), 95 % CI around F1 would be ~±0.10 wide. 6 cells deliver point estimates that match v1 priors.
  • F1 detector is a stub (median + 1.5σ on |Δapf|). Replacing with a tuned detector (Plan 03+) would lift absolute F1 numbers, but iv-relative ordering should remain stable. So v1's iv picks are robust to this.
  • CV at iv=100 ms for workingset is 0.033 — 9× tighter than the strict 0.15 target. There is no plausible analysis where this iv would lose to a slower iv for the steady workload.

What we lose by NOT running full v2

Minor
  • No confidence-interval bars around F1/CV per (workload, iv). Just point estimates.
  • Two iv levels (250, 1000 ms) untested in the v2 sub-pilot; v1's picks at those levels rest on design priors only.
  • Reviewer could ask "what happens at iv=750 ms?" — we can answer from the v1 sweep (pause-fraction known) but not from measured F1/CV.

None of these would flip the v1 iv recommendations. They are completeness gaps for the thesis chapter, not decision risks.

Technical impact
  • Effect-size detection: at n=1 per cell, undetectable effects < ~0.30. At n=3 (full v2), detectable down to ~0.10. We may miss small iv-dependent F1 differences that exist but are tiny.
  • No mixed-effects model can be fit on 6 cells. v2 would have enabled a workload-as-random-effect ANOVA. We forgo that.
  • Plan 03 (window/hop) does NOT depend on full v2 — it uses the trajectory from any captured cells, including the 6 we have. Plan 03 can start now.
Bottom line. The 6 cells confirm the v1 iv recommendation table empirically for the regimes it predicts. Full v2 would strengthen the claim with confidence intervals but is unlikely to change any pick. Defensible to ship v1 as final with a thesis-chapter footnote citing the 6-cell sub-pilot as empirical confirmation.
SECTION 12

Two operator questions

Q1 · How much disk space does a full v2 capture consume?

Short answer

Under 5 GiB total regardless of cell count. That's the entire point of B+3.1.

In the original D-51 design (no streaming), dumps accumulated until cell-end; a single iv=500 d=900s cell would have produced ~445 GiB before D-51 ran. The 90-cell pilot would have needed many TB.

B+3.1 caps disk at 2-3 GiB transient regardless of cell length because the async helper deletes each prev dump as soon as it has computed APF.

Math

Per-cell transient disk (during the run):

  • prev dump · 1 GiB
  • curr dump · 1 GiB
  • helper in-flight on prev · brief overlap up to ~1 GiB
  • Peak: 2-3 GiB per cell at any moment

Per-cell persistent scratch (after cell completes):

  • snapshot_timings.jsonl · ~50 KB
  • apf_trajectory.jsonl · ~50 bytes × n_pairs · ~50 KB max
  • apf_acks/*.apf_done · ~150 bytes × n_pairs · ~150 KB max
  • workload_stderr.log + producer.log · ~50 KB
  • config_timing_experiment.json + heartbeat.json · ~5 KB
  • metrics.json · ~10 KB
  • cell cell_*.json · ~10 KB
  • ~250-500 KB per cell

90-cell pilot persistent total: 90 × 500 KB = ~45 MB. Add manifest CSV (~30 KB), session sentinel (~5 KB), validate report (~50 KB). Grand total persistent: under 50 MB.

Comparison with v1 pilot footprint. v1's 90 cells consumed ~few MB persistent (timing JSONLs only · no apf trajectory · no acks). v2 adds ~40 MB of trajectory + ack files. Both fit trivially in the 100 GiB free disk. The bound that matters is transient: 2-3 GiB peak, never more.

Q2 · Can we run v2 capture and Plan 03 on two VMs in parallel?

Same host?

Don't. Running two VMs that capture-and-pmemsave simultaneously creates exactly the cross-VM contention the multi-VM proposal team already rejected (multi_vm_proposal.html). Each pmemsave writes ~1 GiB to host disk; two simultaneous pmemsaves split bandwidth and host_dt becomes non-stationary. The whole point of Plan 01 (calibrated pause-fraction, stationary host_dt) collapses.

Same-host contention budget
  • CPU · 2 × VM vCPU + 2 producers + 2 async APF helpers + libvirtd · oversubscribed unless host has ≥ 8 idle cores
  • Disk bandwidth · pmemsave throughput already at SSD limit · two concurrent pmemsaves halve each VM's effective rate · effective pause-fraction rises ~2×
  • Last-level cache · workload-A's pages pollute workload-B's cache (subtle but real for memory-bound workloads)
  • libvirt monitor lock · per-domain not shared, so Bug-L not amplified · this one's fine
  • Host page-cache writeback · Bug-G mechanism amplifies (~doubles dirty-page pressure)
Different hosts?

Yes — if and only if each host is independently calibrated. Plan 01's pause-fraction sweep + the 25 ms producer overhead + the iv-cycle math are properties of one specific machine. A second host needs its own sanity / 2a / 2b run before you can trust any iv recommendation from it. ~3 h of host time to validate the new machine.

Calibration checklist for a second host
  • Install bc · libvirt · qemu · numpy · ssh keys (same as current host)
  • Build Phase-2 binaries on the second host's VM image
  • Run sanity (30 s) → confirm guest_dt = intervalMsec + ~25 ms
  • Run 2a (consumer isolation) → confirm host_dt stationary
  • Run 2b (interval sweep at 4 iv levels) → measure new host's pause-fraction curve
  • Update plan02_run.py::PAUSE_FRACTION_BY_IV per host (or make it host-keyed)
  • Then v2 capture or Plan 03 captures can run there independently
Sequential on one host?

Yes, always fine. Run v2 capture on the host, finish, then run Plan 03 on the same host. They use the same calibration; no contention because they run at different times. This is the model you have now.

Recommended ordering
  • v2 capture first (if doing it at all) — produces measured F1/CV per iv per workload
  • Plan 03 second — uses the captured trajectories at the chosen iv to sweep window/hop
  • Plan 03 does not require v2 dumps; it just needs APF trajectories which v2 already produces
  • Skipping v2 and going straight to Plan 03 is also valid — uses v1's iv table + Plan 03's own captured trajectories
Verdict on Q2: two-VM parallelism on the same host stays a no per the multi-VM proposal team. If a second machine is available, calibrate it (3 h) then split workloads across the two hosts. Otherwise, run v2 and Plan 03 sequentially on the current host.
